# 0017 負けた日の稽古場（justLost・敗退翌週の立て直し／掘るか畳むか）

- 日付: 2026-07-06 ／ 数値は全て【仮】 ／ 既存ファイルは読むだけ ／ トーン＝ドラマ主軸（proposals/0009 確定）
- 位置づけ: 単発・選択イベント（justLost 節目）。0003 が担保した「敗退直後の俺の内心一言」の**次の一手＝プレイヤーが手で選ぶ節目**。テーマは「**敗因を掘る（立ち稽古で今日の粗を探す）か／今日は畳む（回復して明日から）か**」。
- 前提（重要）: 本イベントは**単独では発火しない**。トリガの `justLost` が 0003 提案の未配線フラグであること・選択イベントを描く UI 面が現状ゼロであること・効果に既存ヘルパ流用時の衝突があること、の3点を先に解く（→ リスク・注意／レッドチーム A）。

## 狙い

敗退の翌週、稽古場の鍵を開けたところから始める。**熱があるうちに敗因を掘る（能力↑・消耗）**か、**一度畳んで回復する（体力↑・停滞）**か。event_design_v0 設計原則3「正解のない二択（リソース交換・上位互換を作らない）」を守り、明白な最適解を作らない。

- 感情語（悔しい/惨め）を一切書かず、**「鍵」「入口に置いたままの荷物」という所作**と、**この稽古場に固有の物（前の時間の団体が残した番号入りカラーコーン・三番だけ褪せている）**だけで負けた温度を出す（0009 の芯「感情を書くな、感情が起きた"場所の物"を書け」）。
- 熱・執念は「俺」に背負わせない。俺はデッドパン／計算屋のまま、**熱と情は相方の声（相方依存）**へ寄せる（0009 の役割分担）。
- テーマの二極（掘る＝A／畳む＝B）を軸に、**相性軸の第三の脚（C＝場所を変えて相方と）**を足して A/B が単なる体力↔能力の交換に落ちないようにする（下位の三択が別の軸を取り合う＝上位互換不成立）。

## 実装イメージ（既存機構・状態フックに載る形。コードは"提案"・未適用）

### (1) イベントは GameCore の純データ（event_design §5 の型に準拠）

`(id, [選択肢:(ラベル, 効果パッチ)])`。効果は**決定的な state パッチ**（`RandomSource` を受け取らない＝乱数非消費）。

```swift
// 【提案・未適用】GameCore 側：イベントは純データ（テキストはUI層リソース）。
struct EventChoice {
    let label: String
    let patch: (inout GameState) -> Void   // 決定的パッチのみ。RandomSource は受け取らない＝乱数非消費。
}
struct GameEventSpec {
    let id: String                          // "ev_loss_practiceroom"
    let choices: [EventChoice]
}
// 効果例（数値は全て【仮】）:
// A: 掘る   weakSkillOf(センス,発想,表現,華)+2 / mental-2 / stamina-10   ← メンタルは対象外（下記 注意）
// B: 畳む   stamina+15 / mental+2
// C: 飯だけ compat+2 / mental+1 / stamina+5 / money-1500（money<1500 は非活性）
```

### (2) 【要新設・止めて相談は不要（RNG非消費）】決定的パッチを runner の正典 state に永続させる seam が無い

現状 `GameSession.runner` は **private**、`GameSession.state` は `pump()` が summary から**毎回上書き**する（`:142-146`）。よって「選択後に patch を `state` へ適用」しても**次の pump で消える**。永続させるには **runner の権威ある `GameState` に決定的パッチを注入する新 API** が要る（例：`WeekRunner.applyEventPatch((inout GameState)->Void)`・**RandomSource を呼ばない**）。

- これは**乱数非消費＝golden相談は不要**だが、既存 API に無い**新規プラミング**（ドロップインではない）。GameEngine の `applyTraining/applyRest` 等の静的 mutator と同じ純関数流儀で足す。
- 提示は `freeAction` の週頭で `justLost` を見て確定発火 → 選択 → `applyEventPatch` → その後 `choose()` が justLost を落とす。`runner.resolveAction`／`RandomSource` は絶対に呼ばない（preview getter 群と同じ＝golden不変）。

### (3) 【要新設】選択イベントを描く UI 面が現状ゼロ

`monoBox`（`WeekMainView:153-169`）は `Advice` を**1行**描くだけ。`innerVoice` は **name:"俺" を全分岐でハードコード**（`DialogueData:21-39`・確認済）。本イベントは「セットアップ＋2〜3択ボタン＋選択後の話者切替つき会話送り」＝**新規オーバーレイ**が要る。`Advice.name` は nullable で切替の口はあるが、**話者切替つき連続会話のレンダラは未実装**（谷口＝関西弁プール／谷口以外＝標準語プールを相方IDで差し替え・partner_characters_v0）。俺は常に標準語。0003(3) と同一の未実装前提＝本イベントの前提。**UI 変更は swift test で検証不可＝simulator 目視まで込みで「完了」（CLAUDE §D-10）**。

### (4) 【正典同期・必須】balance影響イベントなので sim を回す（golden は別）

本イベントは A で**能力+2（恒久）**を配るのでバランスに効く。

1. `tools/sim_career.py` の `EVENTS_ON`（既定OFF）に本イベントの近似効果を足す。
2. `EVENT_FIRE_CAP=15` の総量予算に数える（`tools/exp_events.py` で優勝率1.4〜1.7%帯の維持を再計測）。
3. 恒久能力の正効果は控えめ（+1〜2）＝§4-B の設計則を遵守（現案は+2で上限）。

**golden（WeekRunner のビット一致）は動かない**。理由は「発生率同期だから」ではなく、**golden の生成器（`gen_golden.py`/`WeekRunner`）がそもそもイベントを走らせない**うえ、パッチが `RandomSource` を一切触らないから。分布は動くので **sim 再計測は必須**（数式・乱数消費順には触れない）。

## リスク・注意

- **【効果バグ・要修正】「弱い能力+2」がメンタルに当たると自壊する**。既存 `weakAbility()`（`WeekMainView:402-405`）は**メンタルを含む5技能**の最小を返す。そのまま流用すると A の「弱い能力+2」がメンタルに当たり、同 A の「メンタル−2」と相殺＝**選択が無効化**する。回避：本イベント専用に**メンタル除外版**（センス/発想/表現/華の4技能から選ぶ）を使う。決定的選択なので RNG は消費しない。
- **【止めて相談】能力を乱数で選ぶ実装にしない**。ランダム選択は event RNG を消費し**乱数消費順／golden に触れる**。決定的（最弱技能）で選ぶこと。乱数に倒すなら**実装を止めてオーナー相談**（消費順の正典・CLAUDE §A）。
- **【前提・最重要】0003 の justLost 配線が唯一のトリガ**。現状 `justLost` フィールドは無い（`justPassedStage` のみ）。`pump()` の `:149-151` で set／`choose()` の `:64` で失効を対称に足す。0003 未実装のまま本イベントだけ載せると発火しない。
- **【前提】選択イベント UI と patch seam が未実装**（実装イメージ(2)(3)）。monoBox は1行・runner は private・state は上書き。ドロップインではなく**新規 UI 面＋新規 state 注入 API**が要る（どちらも RNG非消費＝golden相談は不要だが工数は要る）。
- **【設計上の懸念】確定発火の頻度**。連敗中に毎回この節目が挟まると既読感が出る。`EVENT_FIRE_CAP` 予算に数える／2敗目以降はテキスト差し替え（0003 の single/repeat のように段を分ける）で**演出で解く**（バランスを数値で先に触らない・CLAUDE §B-5）。
- **【要 sim 確認】C の money ゲートと B の支配性**。C は所持金<1,500 で非活性（グレーアウト）。B は無コストで回復値が大きい＝支配的にならないか sim で実測し、支配的なら**掘る/畳む の2択へ畳む**退避案（C を落とす）。
- **声を混ぜない**。俺に執念・鼓舞を持たせない（自己陶酔＝寒い）。熱は谷口／相方へ、俺はデッドパンと事実に保つ（0009・0003 §B）。
- **固有プロップは1つに絞る**。カラーコーン三番の褪せ＝この稽古場固有の一物。天井の染み・冷めたコーヒー等の定番は使わない。増産時に稽古場プロップを乱立させない（希釈で死ぬ）。
- **相方依存の徹底**。谷口以外の相方に関西弁を出さない。話者切替 UI が入るまでは俺の内心のみで暫定運用も可だが、その場合 C（相性軸）の説得力が落ちる（相方の声が本体）。
- **既存作品コピペなし**：全セリフ新規。火花／べしゃり暮らし／M-1実在ネタと各行照合し一致・酷似なし。固有名（コーンの団体名等）は出さない＝架空・匿名。ボケ本文ゼロ・観測／所作／言い回しのみ。

## レッドチーム（A/B）

**A. 実装妥当性**

1. **【確認済】トリガの set/失効点は実在**。`pump()` の敗退分岐 `:149-151`（`lossStreak+=1`）で `justLost` を立て、`choose()` `:64`（`justPassedStage=false`）で対称失効＝既存の justPassedStage と同型で無理なく載る。
2. **【要配線】justLost は現状フィールドすら無い**（`justPassedStage` のみ）。0003 とセットで足す。単独では**発火不能**。
3. **【効果バグ・修正済み案】weakAbility() はメンタルを含む5技能の最小を返す**（`:402-405`）。A の「弱い能力+2」がメンタルに当たると同 A の「メンタル−2」と相殺し選択が無効化。→ **メンタル除外の4技能版**を本イベント専用に用意（決定的・RNG非消費）。
4. **【要新設】決定的パッチの永続 seam が無い**。runner は private・state は `pump()` で summary から上書き（`:142-146`）。patch を session.state に当てても消える。→ `WeekRunner.applyEventPatch`（RandomSource非呼び出し）の新設が要る。golden相談は不要だが**新規プラミング**。
5. **【要新設】選択イベント UI 面が無い**。monoBox は Advice 1行・innerVoice は name:"俺" 固定（`:21-39`）。二〜三択ボタン＋話者切替つき会話送りは**新規オーバーレイ**。simulator 目視まで込みで完了（§D-10）。
6. **【要 sim・トレードオフ健全性】三択が別軸（能力/体力/相性）を取り合い上位互換は不成立**。ただし B が無コストで体力+15/メンタル+2＝回復値が大きく支配的化しないか、C の相性+2 が安すぎないかを `EVENTS_ON` で実測。支配的なら数値調整前にまず 2択化 or 効果量見直し（体感を数値で先に直さない）。
