# 天丼在庫を「腹減ったな」一極から3本に分散する — 俺・審査員側にも三段アークを作る

- 日付: 2026-07-06（夜間リサーチ）
- 位置づけ: 提案のみ・コード改変なし。数値・配置は全て【仮】。既存ファイルは読取のみ。
- 対象の弱点: `writing_craft_and_appeal_v0 §7` が天丼候補として「いけるわ」「今日の最高記録」を名指ししているのに、実際に複数ビートへ仕込まれ回っているのは谷口の「腹減ったな」だけ（`dialogue_design §4` 決勝進出・優勝の感情最大点に配置）。天丼は「1個を3回」でしか機能しない（1回目=フリ／2回目=変化／3回目=最大）のに、在庫が1本深＝他の候補に三段アークが作られていない。谷口偏重で、俺=ツッコミ語り部側・審査員側の再登場語も薄い。

## 狙い

1. **「腹減ったな」以外に、状況を変えて3回反復できる天丼を2本、審査員側に1本の計3本を設計する**。各天丼を必ず〈フリ→変化→最大〉の3ビートに割り付けてから配置する（単発で撒くと全部が「またか」の内輪符牒になって寒くなる）。
2. **谷口偏重を、俺=語り部側の再登場語（「今日の最高記録」「通算/的中率」）と審査員側の再登場語で相殺する**。天丼の"言い回し・トーン"は1回目と固定し、変えるのは「どの場面で・何に対して言うか」だけ（同型維持・状況更新）。
3. **感情最大点（§4 決勝進出/優勝）には必ず各天丼の"3回目"を割り当てる**。最大点は過小申告・言い切らない着地と相性が良く、余白が回収の快感を上げる。ボケ本文は一切書かず、笑いは「観測語の再登場」に逃がす（絶対の一線を守る）。

## どこから学んだか（技術＋出典URL）

- **天丼＝3回で1セット。1回目=フリ／2回目=「来た」と思わせる／3回目=変化を加えて被せる＝最大**。2回目以降は必ず"変化"を足す（単純反復は3回目以降で意外感が減衰）。
  - お笑いと構造 第8回 天丼はどこまで盛れるか: https://satzdachs.hatenablog.com/entry/2020/03/20/033000
  - なぜ天丼は3回までなのか（humorsense）: https://humorsense.s-teem.com/2012/03/post_78.html
- **最強のコールバックは"繰り返し"でなく"再文脈化(recontextualize)"**。同じ語を別の意味・別状況で出し、文脈・言い回し・間のどれかだけ捻る。回収は1回目とトーン・言い回しを揃えると「あのときのあれ」が伝わる。
  - Mastering Callback Humor（comedywriter.info）: https://comedywriter.info/mastering-callback-humor/
  - Writing Callbacks in Your Scripts（Final Draft）: https://www.finaldraft.com/blog/writing-callbacks-in-your-scripts
- **Rule of Three = establish / reinforce / surprise** ／ **SREP = Set-up / Reveal / Escalation / Payoff**。走らせた語は戻るたびに新文脈で大きくなり、最後のPayoffが最大。
  - No Film School: Rule of Three: https://nofilmschool.com/rule-of-the-three
  - BCG Pro: SREP: https://www.comedy.co.uk/pro/inside_track/set-up-reveal-escalation-payoff/
- **「言い切らない／余白を残す」で意外感が増す**（3回目を全部言わずに置くと客が先回りして回収の快感が増す）。同上 Rule of Three。
- **サムさに転ぶ条件＝変化なし反復／内輪符牒化**。天丼候補は"状況説明ゼロでも意味が通る観測語"に限る。
  - 天丼とは（owa-rai お笑い芸人用語集）: https://owa-rai.com/tendon/
- **反復イベントの前イベ/後イベ思想**（同じ素材でもコマンド選択の"前か後か"で意味が変わる／フラグ蓄積で最終回収）。天丼の配置に流用: 1回目=結果ビートの"後"に低温で置き、3回目=最大点ビートの"直前"に置いて回収の呼吸を作る。
  - サクセスモード（Wikipedia）: https://ja.wikipedia.org/wiki/%E5%AE%9F%E6%B3%81%E3%83%91%E3%83%AF%E3%83%95%E3%83%AB%E3%83%97%E3%83%AD%E9%87%8E%E7%90%83_%E3%82%B5%E3%82%AF%E3%82%BB%E3%82%B9%E3%83%A2%E3%83%BC%E3%83%89
  - イベント前後の仕組み（GameWith）: https://xn--odkm0eg.gamewith.jp/article/show/46318
- **審査軸を"比較(T3)＋愛ある辛口(T6)"で言語化する型**（`judge_comments_v1 §0`）。同一審査員が大会をまたいで同じフレーズを返す＝大会構造そのものを天丼の器にできる。
  - M-1グランプリ（Wikipedia・審査軸の参考）: https://ja.wikipedia.org/wiki/M-1%E3%82%B0%E3%83%A9%E3%83%B3%E3%83%97%E3%83%AA

## 実装イメージ（そのまま貼れるスニペット／既存構造に載る形・提案であって差分ではない）

### 天丼A「今日の最高記録」＝俺の"最小反応を計測する"温度差実況（過小申告基調）

同型: 俺が客席の"最小の反応"を「今日の最高記録」として淡々と計測する。**計測する対象だけが状況ごとに変わり**、最大点で計測癖そのものが追いつかなくなって回収。谷口ではなく**俺=語り部**が背負うので、谷口偏重を直接崩す。

| 回 | 配置（§4連動・前/後イベ） | 状況差分 | 「最高記録」の中身（＝変えるところ） |
|---|---|---|---|
| 1回目(フリ) | 序盤の小ライブ後・**後イベ**・低温 | 日常・不作 | ネタと無関係な些末な反応を1つ計測（例: 相方が段を踏み外した所で客が一人だけ動いた） |
| 2回目(変化) | 中盤・敗退/不作ビートの後 | 逆境 | 計測対象がさらにズレる（審査員がペンを"置いた"理由が分からないまま記録に数える） |
| 3回目(最大=決勝進出/優勝) | 感情最大点の**直前 or 直後** | 本番の頂点 | ついに"記録"が本物の歓声になり、計測が追いつかない＝過小申告で言い切らず着地 |

```
// dialogue_design §6 テーブル行形式: (場面ID, 相性帯, 条件フック, 技法タグ, 本文)  ※全て【仮】
(小ライブ結果, 任意, 天丼A/stage=1・低温後イベ, 温度差実況+過小申告,
  地の文(俺)=「今日、客席で動いたのは一人だけだった。しかもそれは、谷口が段を踏み外したところで。ネタと無関係の、今日の最高記録だ。」)

(大会敗退, 任意, 天丼A/stage=2・敗退後イベ, 温度差実況,
  地の文(俺)=「反応はなかった。正確には、審査員が一度だけペンを置いた。書き終えて置いたのか、書かないために置いたのかは分からない。分からないまま、今日の最高記録に数えておく。」)

(決勝進出決定 or 優勝, 任意, 天丼A/stage=3・最大点, 過小申告+言い切らない,
  地の文(俺)=「今日は、数えるのをやめた。……いや、数えようとして、追いつかなかった。最高記録の付け方が、たぶん、もう分からない。」)
```

### 天丼B「いけるわ」＝谷口の根拠なき断定 × 俺の"通算成績/的中率"検証（温度差実況の核）

谷口は3回とも**同じ「いけるわ」（同型維持）**。変わるのは"賭けの大きさ"と、**俺が持つ再登場語「通算/的中率」**だけ。俺側に検証カウンタを持たせることで、谷口の口癖に"対の観測語"が生まれ、決勝で反転して最大化する。

| 回 | 配置 | 谷口（断定・不変） | 状況差分 | 俺=語り部の再登場語（検証） |
|---|---|---|---|---|
| 1回目 | 序盤・平場（低い賭け） | 「いけるわ」 | 平場 | 「的中率は、今のところ低い」（外れ） |
| 2回目 | 中盤・格上と同卓/賭けを上げる | 同じ「いけるわ」 | 上げる | 「通算、五分に戻した」（半分当たり＝検証の中間） |
| 3回目(最大) | 決勝直前・**前イベ** | 同じ「いけるわ」 | 本番の頂点 | 「今回のは、当たりだ。通算成績は、聞くな」（過小申告で認める＝回収完成） |

```
// DialogueData.swift に足す想定の在庫（提案・数値/条件は【仮】）
// 谷口=関西弁・楽観／俺=標準語・計算屋。stageは順序固定で1回ずつ発火。
(場面=稽古上がり or 大会前, 天丼B/stage=1, 過小申告(谷口)+ズレた冷静さ(俺),
  谷口=「これ、いけるわ。」／俺=「どこがどういけるのか、根拠は今日も無かった。的中率、現状かなり低い。」)

(場面=格上と同卓/賭けを上げた週, 天丼B/stage=2, 温度差実況,
  谷口=「……いけるわ。」／俺=「同じ台詞、同じ顔。ただ今日は半分当たった。通算、五分に戻した。」)

(場面=決勝直前(前イベ), 天丼B/stage=3, 過小申告+言い切らない,
  谷口=「いけるわ。」／俺=「……今日のは、当たりだ。通算成績は、聞くな。」)
```

### 天丼C 審査員の再登場語＝紙講評を大会またぎで返す「まだ、行儀がいい」（毒は丁寧な三段オチ）

`judge_comments_v1 §0` の **T3比較型＋T6愛ある辛口**。同一審査員が予選→上位回戦→決勝で同じ査定語を返し、**3回目で落ちの向きが"毒"から"承認"へ反転**する（丁寧な三段オチの型は保つ）。大会構造そのものが天丼の器になる。

| 回 | 配置 | 審査員の再登場語（同型） | 状況差分と三段オチの落ち |
|---|---|---|---|
| 1回目 | 予選（敗退寄り紙講評） | 「まだ、行儀がいい」 | 低評価寄り。丁寧→丁寧→軽い毒で締める |
| 2回目 | 上位回戦（通過） | 同じ「行儀」語 | 中評価。前回との差分に触れ、毒が一段軽くなる |
| 3回目(最大=決勝) | 優勝講評 | 同じ「行儀」語 | 最上級へ反転。落ちが承認になる＝毒の期待を裏切って着地 |

```
// judge_comments_v1 §1 の1行形式（紙講評）。※同一コンビ・同一審査員で3大会をまたぐ天丼フラグ付き【仮】
[天丼C/stage=1・予選敗退寄り・T1+T6]
  「上手い。ただ、まだ行儀がいい。教科書の角を、一枚も折っていない。」
[天丼C/stage=2・上位回戦通過・T3比較]
  「去年より行儀が悪くなった。褒めています。折れた角が、一枚。」
[天丼C/stage=3・決勝最上級・反転(三段オチの落ちを承認へ)]
  「相変わらず上手い。相変わらず……いや、今日は違う。行儀の話は、もうしない。」
```

### 天丼を"順序固定・1回ずつ"で発火させる器（提案・GameState はすでに Codable）

```swift
// 提案（GameState へ足す想定・数値/命名は【仮】）: 天丼の進行だけを持つ最小カウンタ

// 発火規則（提案・配置ロジックはUI/選択層で）:

//  ・stage=1 は「結果ビートの"後"（後イベ）」に低温で置く。
//  ・stage=3 は「感情最大点ビートの"直前"（前イベ）」に置き、回収の呼吸を作る。
```

## リスク・注意

- **【設計上の懸念】"1回目がスベったまま3回目に行く"共倒れ**。stage=1 は必ず"軽く成立"させること（過小申告で外さない語＝「最高記録」「的中率」「行儀」はいずれも状況説明ゼロで意味が通る観測語なので比較的安全）。スベったまま繰り返す運用は禁止。
- **谷口の「腹減ったな」と近接させない**。同一週・同一大会の最大点に天丼を2本重ねると密度過多で全部が寒くなる。§6 の「腹減った＝最大点」と、天丼A/Bの stage=3 は**別の周回・別の最大点に振り分ける**か、最低でも `tendonMinGapWeeks` 以上空ける。1周内は「腹減った＋他1本」程度に抑える。
- **天丼Cは"同一コンビ×同一審査員が3大会またぐ"追跡が要る**。予選は審査員非表示（`judge_comments §1`）なので、Cの stage=1/2 を"どの審査員の声か明かさずに紙で返し、stage=3(決勝)で顔が割れて回収"にすると、非表示仕様と矛盾せず、むしろ「あの紙、あの人だったのか」の伏線回収になる。ここは要オーナー確認。
- **4回目を作らない・言い切らない**。stage=3 は全部言わず余白で置く（過小申告と最大点は相性が良い）。ボケ本文には一切踏み込まない（絶対の一線）。
- **数値は全て【仮】**。`tendonMinGapWeeks`/`tendonPerYearCap` は sim＋実機目視（`writing_craft §7-B`）で詰める前提。バランスは最後、まず配置と筆致で解く。

## レッドチーム（2軸）

- **体感で寒い/くどい方向**: 3本を同時期に走らせると"天丼だらけ"に見えて内輪化する。→ 反証: `tendonPerYearCap=1` で「腹減った＋他1本」に制限し、A/B/C は別周回に薄く分散。同型維持・状況更新を守れば"またか"でなく"あのときのあれ"になる（callback研究）。それでも寒いなら stage=2 を削って2ビート運用に落とす退避策あり（ただし天丼の効きは弱まる）。
- **実装/仕様と衝突する方向**: stageカウンタの追加は GameCore の状態を増やし、**golden（乱数ビット一致）に影響しないか**が懸念。→ 反証: `tendonRecord/Ikeru/Gyogi` は**乱数を消費しない純表示フラグ**で、発火判定は結果ビート確定後の分岐に置く（RNG非消費の正直getter方針＝`ui_redesign_pawapuro_uma_v0` と同じ思想）。消費順に触れなければ golden は無傷。実装時は「乱数を引かない」ことを `swift test` で必ず確認（CLAUDE.md §A-2）。
