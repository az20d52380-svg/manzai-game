# 改善提案: 常時表示の心の声（平常プール）の資産化 v0

対象ジャンル: 漫才師コンビ育成SLG（iOS/SwiftUI・本編10年・週送り）。数値は全て【仮】。提案であって既存ファイルは未編集（読むだけ）。

## 狙い

fun_flow_review が指す「4〜7年目の停滞レジーム」の**地色**を厚くする。§1 は中だるみ帯で「良い週送りで送る中身が薄い」ことを弱点に挙げ、§3 は年1回の一言ラダー（谷口/支配人・関西弁）を別レイヤーとして足す。本提案はその**手前**、毎週必ず出る俺の平常モノローグを対象にする。writing_craft §0 が言う通り、コメディを"書かない"本作では地の文の筆致が唯一の差別化点で、そこが平凡なら普通の育成ゲーム止まりになる。

その最頻テキストが実装では平常3行の平坦文（`今週も、一歩だけ前に。` 他 L58-62）で、writing_craft の8技法も基調「過小申告＋温度差実況」も一切載っていない。judge135項・dialogue110本を投じた筆致資産化が、**最も多く読まれるこの一枚だけ素通りしている**。

> レッドチーム反映: 元案の「fun_flow が二度名指しした一枚」は誇張。§1/§3 は 平常プールを名指ししていない（§3 は別人物・別頻度の年次ラダー）。ここは「fun_flow が指す中だるみ帯の毎週の地色」と正確に言い直す。warrant を盛らない。

狙いは三つ。
1. **量**: 平常プールを最低30本／目標90〜120本規模へ（後述B）。※content_inventory §2 は「1年MVPは全カテゴリ充足・量産不要」だが、平常は e（年内発火頻度）が突出して高い**唯一の例外帯**。だからMVP内でも carve-out として増やす価値がある、という限定した主張（下でeを詰める）。
2. **差分化**: 状態が「平常」で固定でも、拾う"観測レンズ"を替えて別角度から実況する（Disco Elysium型）。単純modを直近回避付きに替え、少週での即ループを止める。
3. **隔離**: 天丼（口癖・小物再登場）は最頻の平常プールに入れず低頻度枠へ退避する。最頻の場所で天丼を回すと近接反復で即寒くなる（writing_craft §2-7）。

数値バランスは動かさない（規律B-5）。演出・頻度・テキストのみ。

## 必要本数（content_inventory §2 の N/e 式を平常に適用）

式: 直近ID禁止が効く前提で、反復間隔 ≈ N/e。ここで **e＝平常が1年（48週）で発火する回数**。content_inventory の e は「複数年の年あたり遭遇頻度」だが、平常は**1回のプレイ内でも1年間に何十回も鳴る**ので、単位を「年」でなく「1プレイの48週」で見るのが正しい（元案は年単位のまま6週結論を出しており、単位が混線していた＝ここを直す）。

- e（48週中の平常週）: 低体力・金欠・連敗・通過・大会前の高優先が抜けた"平常週"は、停滞層ほど多く **1年あたり20〜30回**規模。
- 現状 N=3・直近回避なし・salt=`fame+compat+stamina`（停滞週はほぼ不動）→ 実質2〜3週で一巡し、悪くすると同一値張り付きで**同じ1行が連続**。fun_flow の「少週で即ループ」を裏付ける。
- 1年反復させない下限: **N ≥ e ≈ 25〜30本**。
- 体感許容（1プレイで飽きさせない）: **N ≥ e×4 ≈ 90〜120本**。
- 結論: 平常プールは**最低30本／目標90〜120本**。judge135・dialogue110と釣り合う投資対象は、最も読まれるこの一枚。

> レッドチーム反映: **平常だけ直しても隣が崩れる**。同じ innerVoice の `lowStamina`／`lowMoney`／`losing`／`passedLines` は**各2行**で、停滞層は低体力・金欠・連敗に頻繁に入る。2行プールは平常より速く一巡する（例: `losing` は salt=lossStreak で2行を往復するだけ）。**直近回避と増量は平常と同時に4状態プール全部へ効かせる**のが本筋。平常のみのスコープは局所最適。

技法タグ配分（writing_craft §2-1「1テキスト＝技法1つ」「盛ると寒い」厳守）: **過小申告＋温度差実況を7〜8割の主軸**、誇張・意外な直喩・擬人化・自虐・ズレた冷静さは**各1割以下のアクセント**。タグ×観測レンズをクロスさせると、同本数でも組合せが増えて反復体感が下がる（Hadesの低コスト版）。

## 実装イメージ（既存構造に載る形／レッドチーム反映版）

### (1) 直近回避 ＝ 「週インデックス駆動」で行う（元案の static var を破棄）

**元案の欠陥（レッドチームで確定）**: 呼び出し元 `monoBox` は SwiftUI の computed view var（WeekMainView L153-156）。body はスクロール・無関係タップ・再レイアウトのたびに何度でも再評価される。`static var recent` に append する方式だと、**同じ週の中で心の声が redraw ごとにチラつき・入れ替わる**。「直近回避」は"1週=1回の印象"を仮定しているが、実際の呼び出し回数はUIが決める。これは体感バグ。

加えて元案スニペットは `recent` を全プール共有していたため、`平常`(3行)と`losing`(2行)が同じ履歴で相互汚染する二次バグもあった。

正しい設計は **選択を「その週で不変な値」の純関数にする**こと。golden への影響は無い（下記確認）ので、素直に週インデックスで決める。

```swift
// 提案: 直近回避を「週番号」駆動にして body 再評価に対して不変にする。
// weekIndex は GameState 側の単調増加値（週送りでのみ変わる）。
// recent は「表示履歴」＝GameState/セッション側に持たせ、週送り時にのみ更新する。
// これで (a) 同一週内は何度描画しても同じ行 (b) 週をまたぐと直近回避が効く。
static func pick(_ pool: [String], weekIndex: Int, recent: [String]) -> String {
    guard !pool.isEmpty else { return "" }
    let avoid = min(recent.count, max(0, pool.count - 1))   // プール枯渇で詰まないよう上限
    let live  = pool.filter { !recent.suffix(avoid).contains($0) }
    let src   = live.isEmpty ? pool : live
    let i     = ((weekIndex % src.count) + src.count) % src.count
    return src[i]
}
// recent への追加は「週送り確定時」に1回だけ、GameState 側で行う（描画では触らない）:
//   session.recentMono.append(chosen); if session.recentMono.count > 8 { removeFirst() }
```

**golden／正典への影響**: 無い。確認済み — `DialogueData`／`innerVoice`／`平常` は `ManzaiGame/`（`WeekMainView`・`CommandData`）からのみ参照され、`GameCore/` にも `tools/gen_golden.py` にも一切現れない。表示テキスト選択は**UI層で完結し、乱数消費順（CLAUDE.md §A-2）にも golden にも載っていない**。したがって元案の「static可変が golden一致を壊す」という懸念は**過大**だった。真の害は上記の「redraw churn」と「プール共有」であって、正典破壊ではない。とはいえ recent を GameState に置くのは正しい（決定的ロジックと表示履歴の分離＝セーブ/ロードで心の声が飛ばない利点もある）。

> レッドチーム反映: 元案のレンズ名 `最悪想定` は**サンプル本文が一つも最悪想定になっていなかった**（下記旧line3/line8は計算屋・擬人化で、心配性の catastrophize ではない）。レンズは差分化エンジンの核なのに、3枠中1枠が中身空で"仕組みだけ立派"だった。→ レンズ名を心配性の実体に合わせ `心配` に改め、**本当に最悪を数える過小申告の行**を配属する（下B参照）。また `絶対値()` は Swift でないので `abs`／剰余で表記し、`平常[lens]!` の force-unwrap は既定へフォールバックさせて落とさない。

### (3) 平常プールの原案（俺＝標準語・技法1つずつ・架空・ボケなし・【仮】）

行式 `(場面ID, レンズ, 技法タグ, 本文)`（dialogue_design §6準拠）:

**過小申告（主軸・多め）**
- `(平常, 損益, 過小申告, 「特に何もない一週間だった。芸人にとっては、それが一番こわい。」)`
- `(平常, 損益, 過小申告, 「昨日書いた三行のうち、一行は今朝、黙って消した。差し引き、たぶん前進だ。」)`
- `(平常, 心配, 過小申告, 「今のところ、何も起きていない。何も起きていない時ほど、俺は次に何が起きるかを数え始める。」)`  ← 旧line3を差し替え。心配性が"最悪を数える"実体を入れ、レンズと本文を一致させた。

**温度差実況（谷口を横目で観測・事実だけ置く／谷口の本音は言語化しない）**
- `(平常, 谷口観測, 温度差実況, 「谷口は寝起きに冷蔵庫の前で三分立っていた。何を考えていたかは、たぶん昼飯のことだ。」)`  ← 旧line4「いける気がすんねん」を差し替え。理由: それは writing_craft §2-7 が天丼素材に指定する「いけるわ」の近縁変種で、最頻プールに入れると隔離方針と自己矛盾する。大食い×温度差の別ビートに置換。
- `(平常, 谷口観測, 温度差実況, 「谷口はもう昼飯の話をしている。俺はまだ、先週の客席の音を思い出している。」)`
- `(平常, 谷口観測, 温度差実況, 「谷口が『腹減ったらええネタ出えへんやろ』と真顔で言った。反論の材料が、俺には一つもなかった。」)`  ← 谷口の引用は関西弁のまま、地の文の「言うた」を俺=標準語の「言った」に修正（口調ブレ潰し）。

**ズレた冷静さ・自虐（低頻度）**
- `(平常, 損益, ズレた冷静さ, 「冷静に今週の勝算を数えた。増えても減ってもいない。それを進歩と呼ぶかは、来週の俺に任せる。」)`

**擬人化・意外な直喩（最低頻度アクセント／要プレイテスト）**
- `(平常, 心配, 擬人化, 「ネタ帳の余白は、まだ半分しか俺たちを信用していない。」)`  ← 残すが"最初に切る候補"。詩的に寄ると可笑しみでなくキザに転ぶ（§2-1「盛ると寒い」）。実機で義務タップ側に感じたら真っ先に落とす。

いずれも反応・観測・言い回しのみでオチは書いていない（絶対の一線遵守）。技法は各1本1つ。天丼素材はここに入れない。俺の地の文は全て標準語、谷口の引用のみ関西弁。

## リスク・注意

- **直近回避は「週駆動・GameState保持」で実装する**。static可変＋描画時appendは redraw でチラつく（対象箇所の `monoBox` が view var のため）。recent を GameState に置くと (a) 描画不変 (b) セーブ/ロードで履歴保存 (c) ロジック=決定的／表示履歴=状態、の分離が同時に得られる。golden には無関係（Dialogueは正典外＝確認済み）。
- **平常だけでなく4状態プール（低体力/金欠/連敗/通過）を同時に手当てする**。各2行は平常より速くループする。増量と直近回避はまとめて。
- **レンズ `心配` に必ず"最悪を数える過小申告"を実配属する**。空レンズは差分化を偽装するだけ。
- **量産は"技法1つ・盛らない"が命**（writing_craft §2-1）。90〜120本を急ぐと過剰装飾で寒くなる。基調7〜8割を過小申告＋温度差に固定し、誇張・直喩は薬味に留める。擬人化・直喩の行は"最初に間引く枠"と割り切る。
- **天丼の隔離を守る**。「いけるわ」「今日の最高記録」等の口癖・小物を平常に混ぜない。低頻度枠（結果画面・大会前）へ置き、再登場間隔は GameConfig【仮】。
- **MVPスコープとの整合**: content_inventory §2 は「MVPは量産不要」。平常はその**明示的な例外**（毎週発火でeが突出）として扱い、他カテゴリの量産前倒しには広げない。
- **効果判定は机上でなく実機**（CLAUDE.md §10・writing_craft §4）。停滞層の4〜7年目相当を周回し「タップして送った時に可笑しいか／義務タップ化していないか」を目視。fun_flow §3の年次一言ラダー（年1回・谷口/支配人・関西弁）とは別レイヤー（あちらは年1回の事件、これは毎週の地色）なので、両方入れて口調・頻度が干渉しないか確認。
- 本提案はMVP未実装箇所の拡充であって、既存の平坦プールを欠陥断定するものではない（"未実装なだけ"）。

## レッドチーム（A/Bで見つけた問題と対処）

**A. 実装妥当性**
1. **【最重要・修正済】golden破壊の誤診 → 真の害はUI churn**。元案は static可変 recent が「同シード再走で golden がズレる」ことを最大の懸念にしていた。確認の結果 `DialogueData/innerVoice/平常` は `GameCore/`・`tools/gen_golden.py` に一切参照が無く、表示選択は正典外で golden に載らない。真の欠陥は別で、呼び出し元 `monoBox`（WeekMainView L153-156）が SwiftUI の computed view var のため、body再評価ごとに `pick` が走り、static append 方式だと**同一週内で心の声がチラつく**。対処: 選択を weekIndex の純関数化＋recent を GameState 保持に変更（本文(1)）。
2. **【修正済】プール共有バグ**。元案の `recent` は全プール共有で、平常(3行)と連敗(2行)が相互汚染し `avoid` 計算が小プールで詰まる。対処: recent は状態別・上限付き、`live.isEmpty` フォールバックで詰み回避。
3. **【修正済】スコープが局所的**。平常のみ厚くしても隣の2行プール（低体力/金欠/連敗/通過）がより速くループする。対処: 同時手当てを明記。
4. **【修正済】レンズ `最悪想定` が空**。差分化の核なのに3枠中1枠に該当本文が無く、仕組みだけ立派で中身が伴っていなかった。対処: `心配` に改名し実本文を配属。
5. **【修正済】force-unwrap / 非Swift表記**。`平常[lens]!`・`絶対値()` は「そのまま貼れる」要件（規律⑤）を満たさない。対処: フォールバック＋`abs`/剰余に。
6. **【修正済】単位混線**。N/e を「年」単位のまま6週結論に使い、content_inventory の年次モデルと平常の週次実態が混ざっていた。対処: eを「1プレイ48週の発火回数」と再定義。
7. **未確定依存の明示**: weekIndex／recentを載せる GameState・session の具体フィールドは未確認（本提案は差し込み口の設計まで）。実装時に GameState 形と週送り確定タイミングへ配線する必要がある＝提案段階。

**B. 会話の中身**
