# 提案 v0 — 「慣れの外し方」相性15-20（ツーカー帯）固有の選択イベント

- 日付: 2026-07-06 ／ 数値は全て【仮】／ 既存ファイルは読むだけ ／ トーン＝ドラマ主軸（proposals/0009 確定）
- 対象読者: イベント機構（event_design §2 二択）実装担当・DialogueData / WeekMainView の書き手
- 位置づけ: 単発選択イベント #40 相当。既存の突発イベント群（event_design §1-D「コンビの機微」）は相性**下**（8未満・楽屋無言 #18 等）に厚く、相性が上がりきった後（15-20 ツーカー帯）にプレイヤーが握る選択が空白。その明示 gap を、相性帯 15-20 を hook にした二択1枚で埋める。
- **【最重要・実装前提】** レッドチームで確認した通り、**二択イベントの発火/画面/効果適用の基盤が GameCore にもアプリにも無い**（`WeekRunner` の Phase は `freeAction`/`tournamentDecision`/`gpRound` 等のみ・相性帯フックも無い）。本提案は「既存機構に載る」ものではなく、**イベント基盤を建てる最初の1枚**。基盤が触る乱数消費順＝要相談（→ 実装イメージ(4)・レッドチーム A-1/A-2）。

## 狙い

相性は上げるほど良い、が現状の暗黙の設計。だが下積みの実感では「通じ合いすぎた二人」に固有の落とし穴がある——目配せだけで一本終わる＝客より先に自分たちがオチを知っている予定調和。ここに「安住（完成された呼吸を固める）か、破壊（一度ほどいて伸ばし直す）か」という、**相性が高いからこそ発生する意思決定**を置く。

やることは一つ。**相性 15-20（ツーカー帯）に初めて入ったら、「あえて崩すか／固めるか」の二択を一度だけ出す**。要点は三つ。

1. **正解のないトレードオフ**（event_design §2・設計原則3）。A=崩しは**センスを伸ばす代わりに相性を一度ほどく（後退）＋体力コスト**、B=固めは**表現を確定で積むが伸びしろは動かない（安定・上限低め）**。上位互換を作らず、期待値でなく"どういうコンビでいたいか"で選ばせる。
2. **相性帯 15-20 を発火条件そのものに使う**。既存イベントが手薄な"上がりきった後"の帯に固有の顔を付ける。※`config.compatCap == 20`（GameConfig.swift:83 で確認）＝谷口に阿吽 21+ 帯は出ないので、15-20 がツーカー帯の上限＝境界どおりに収まる（別帯への漏れが構造上ない）。
3. **相方依存の声で"通じすぎ"のドラマを出す**（0009）。谷口＝関西弁・先回りの一言（ツーカー帯の温度）。谷口以外の相方は標準語（声は相方 ID で差し替え）——**ただし現状 partner_id は未実装＝MVP は谷口一択**（下記 発火条件・レッドチーム A-3）。

## 発火条件（実フック）

- **一次条件**: 相性が **15-20（ツーカー帯）**。**【設計上の懸念・降格】** エンジンに相性帯フック（0-7/8-14/15-20 の帯導出）は**存在しない**（コード上の相性しきい値は `JudgeData.swift:57` の `compat >= 18` のみ）。よって本イベントは「既存フックに載る」のではなく、**帯導出の追加を伴う**。しきい値はイベント側にハードコードせず `config.compatCap` 等に集約する（GameConfig が正本・開発規律 B-4）。
- **トリガ（推奨＝確定・非RNG）**: **「相性が初めて 15 に到達した週」に一度だけ確定発火**（大会週を除く）。**週次12%のランダム抽選には載せない**——理由は下記。同一キャリア1回（再発させない＝予定調和化を避ける）。「初到達」の記録は `GameSession` に非RNGの `didFireTsuukaChoice: Bool` 相当を持たせる（`justPassedStage` と同型の表示用 state・runner の乱数に触れない＝golden 不変）。
  - **なぜランダムでなく確定か（0010 との差の要点）**: 週次12%抽選そのものが**未実装**。載せるには runner に新規ロールを差し込むことになり、**乱数消費順が動いて golden（3年ビット一致）が壊れ得る**（→ 実装イメージ(4)・レッドチーム A-2）。「帯初入で確定」なら**新規 RNG を1回も引かない**ので golden を守れる。
- **MVP（1年）での発火**: 相性 15 到達は**終盤寄り**なので、MVP では**終盤到達で1回発火**する想定。稽古で相性を積めば1年内に届く（本編でも同条件で発火可）。早期暴発は「初到達1回きり」の性質で自然に抑止。
- **専用アークとの境界（重複回避）**: 各相方の専用アークも相性 15 開始（partner_characters §・谷口「なんとかなるの研究」等）。本イベントは帯共通の汎用枠。**【設計上の懸念】** 専用アーク state はコード未実装（grep 一致なし）＝排他の参照口が無い。**当面は谷口単独・本イベントを先に置き、谷口アークが実装される時に発火週の前後を1コミットで調停する**（同じ「相性が高い二人」ネタの二重発火を避ける・段階投入）。谷口以外への解放は partner_id 実装後（本編）。

## イベント本文（場面のセットアップ＋二択）

> トーン＝ドラマ主軸。感情語（悲しい/悔しい/惨め）を地の文から抜き、**"この稽古場・この相方に固有の少しズレた物"**（＝二人にしか読めなくなった段取りの略字）に温度を背負わせる。俺＝標準語・計算屋・自己憐憫なし。**会話は話者ごとに1セリフずつ（"／"で詰めない）**。ボケ（オチ/ネタ本文）は書かない。数値は全て【仮】。

**セットアップ（地の文・俺視点）**

```
稽古場のホワイトボードに、三本目の段取りが消されずに残っている。
いつからか、二人にしか読めない略字になった。「A→タ→落」。
書いた俺と、読める谷口。それで足りている。
今日も、目配せだけで一本通した。オチの場所を、客より先に二人が知っている。
```
- 技法: **固有プロップ**（"二人にしか読めない略字"＝この相方・この稽古場固有の少しズレた物・定番プロップでない）＋**デッドパンの一滴**（「書いた俺と、読める谷口。それで足りている。」＝真顔の事実の並置・自己言及なし）。谷口以外の相方版では下の谷口セリフを標準語へ差し替え（→バリエーション）。旧稿の「解読できる人間は、この世に二人しかいない」は誇張気味＝**キザ回避のため削除・差し替え済**（レッドチーム B-2）。旧稿にあった地の文と俺セリフの"客より先にオチを知る"の**二重掲出は解消**（俺の直後セリフを削る・B-1）。

```
谷口「最近、俺ら目ぇ合わすだけで一本終わるな」   ← 谷口：先回り観測（ツーカー温度・関西弁）
俺「終わる。」                                   ← 俺：デッドパン（事実処理・肯定を最短で）
谷口「気持ちええけど、こわいやつやな、それ」     ← 谷口：先回り観測（関西弁）
```

---

**選択肢A：あえて崩す実験に出る（一度ほどく）**

**【効果】** センス +2 ／ 相性 −1（合いすぎた呼吸を一度ほどく・後退）／ 体力 −10。※ **新規 RNG 非消費・確定効果**（→ 実装イメージ(4)）。数値は全て【仮】。

```
俺「一回、崩す。二人にだけ通じる呼吸は、劇場の後ろまで届かない。」   ← 俺：計算屋の事実（宣言でなく客観・キザ回避に改稿）
谷口「……やろな。お前がそのボード消すの、そろそろやと思とった」     ← 谷口：先回り（相手の意図を言い当てる・関西弁）
```
- 改稿点（B-2）: 旧俺セリフ「俺たちにしか効かない呼吸は、俺たちの外に出ない」は"俺たち"重複＋詩的宣言＝キザ寄り。**「劇場の後ろ（＝客席後方）まで届かない」という計算屋らしい具体の事実**に差し替え。

**選択肢B：完成した呼吸を固める（安定）**

**【効果】** 表現 +2 ／ 相性 +1。**伸びしろ（センス・発想）は動かさない**（安定だが上積みの天井は低い・上振れも下振れもない）。※ 新規 RNG 非消費・確定効果。数値は全て【仮】。

```
俺「今は固める。崩すのは、固まってからでいい。」            ← 俺：計算屋の判断（順序の事実）
谷口「ええよ。……この略字、もうちょい二人で使い倒そか」    ← 谷口：先回りの受け（固有プロップの回収・関西弁）
```

**相方依存バリエーション（谷口以外の相方＝標準語ベース・声は相方 ID で差し替え／※現状 partner_id 未実装＝本編用スキャフォールド）**

```
地の文の谷口セリフ→標準語: 「気持ちいいけど、ちょっと怖いな、それ」
A・相方「……だろうね。きみがそのボードを消すの、そろそろだと思ってた」
B・相方「いいよ。……この二人だけの合図、もう少し使い倒そう」
```
- 谷口以外に関西弁を混ぜない（相方依存の声・0009）。ツーカー帯 15-20 なので相方は"先回りして俺の意図を言い当てる"温度で統一。旧稿の「消したがってるの。」の語尾は柔らかすぎ＝中性化（B）。※`compatCap == 20`＝谷口に阿吽 21+ は出ない＝この帯が谷口の上限。

## どこから学んだか（構造＋出典 URL）

学んだのは**構造・型のみ**。既存作品のセリフ・攻略本文は転載しない。

- **育成 SLG の"パラメータ上限帯に固有イベントを置く"設計**（低帯・中帯・高帯で別イベントを撃ち分け、上げきった後にも選択を残す二層構造）。相性の高帯にだけ出す選択がこの型。 https://ja.wikipedia.org/wiki/実況パワフルプロ野球_サクセスモード
- **上位互換を作らないリソース交換（正解のない二択）** — A は「センスを買うのに相性を売る」、B は「表現を確定で買うが伸びしろは買えない」。期待値で割り切れない＝計算屋でも最適解を出せない（event_design §2 の核）。 https://ja.wikipedia.org/wiki/トレードオフ
- **予定調和・ルーティン化がパフォーマンスを頭打ちにする（あえて崩す＝脱馴化）**。慣れきった手続きを一度壊すことで再び伸びる、という運動学習/即興の知見が「A＝崩して伸ばし直す」の裏付け。同時に"崩さず磨く熟達"も別の正解＝非対称のトレードオフ。 https://ja.wikipedia.org/wiki/インプロヴィゼーション
- **通じ合いの符牒化（イングループの内輪言語）** — 親密な二人だけに通じる略号・合図が育つ現象。"二人にしか読めない略字"の固有プロップの下敷き。 https://ja.wikipedia.org/wiki/隠語
- **感動を宣言しない（事実と物で置く）** — "通じすぎ"の重さを地の文で叫ばず、"二人にしか読めない略字"一つに背負わせる（0009 の芯）。 https://ameblo.jp/isawo-t1307/entry-12689913076.html
- 自作 `event_design_v0` §1-D・§2・§設計原則3・§4-B（正解のない二択・上位互換禁止・EVENT_FIRE_CAP=15）、`partner_characters_v0`（専用アーク=相性15開始／谷口=関西弁）、`proposals/0009`（ドラマ主軸・固有の物・相方依存の声）、`proposals/0010`（節目ゲートの型・"確定効果を先に・RNGは相談後"の段階投入）。

## 実装イメージ（既存機構に載る形・コードは"提案"）

**(1) まず二択イベント基盤が要る（前提の是正）。** レッドチーム A-1 の通り、`WeekRunner` に二択 Phase は無く、アプリに選択イベント画面も無い。本イベントは**基盤の最初の実装対象**。効果対象自体は既存6軸＋体力の増減のみ＝**効果に新システムは不要**だが、**発火判定・二択提示・効果適用の口は新設**になる。

```swift
// 【提案・未適用】ツーカー帯イベント #40。二択イベント基盤の第一号として実装する想定。
// 発火: 相性が初めて 15 に到達した週（帯初入）に一度だけ確定。週次ランダム抽選には載せない。
//       → 新規 RNG を一切引かない = 乱数消費順に触れない = golden 不変（(4)）。
// 効果は既存パラメータの確定増減のみ。A/B とも RNG 非消費。
struct FamiliarityUnlearn {
    // A: 崩す
    static let senseUpA   = 2    // 【仮】※event_design §4-B の恒久正効果+1〜2 上限・EVENT_FIRE_CAP=15 の予算内
    static let compatDnA  = -1   // 【仮】呼吸を一度ほどく（後退）
    static let staminaDnA = -10  // 【仮】
    // B: 固める
    static let expressUpB = 2    // 【仮】
    static let compatUpB  = 1    // 【仮】伸びしろ(センス/発想)は不動
}
```

**(2) 相性帯の判定は新設・しきい値は config に集約。** 0-7/8-14/15-20 の帯導出はエンジンに無い（既存は `JudgeData` の `compat >= 18` 単発のみ）。帯導出を1関数として追加し、境界値は `config.compatCap`（=20）等から引く＝イベント側にハードコードしない（帯の正本は1箇所・開発規律 B-4）。`compatCap == 20` なので 15-20 が最上帯＝谷口で 21+ 帯の死に枝は生まれない。

**(3) 発火記録は非RNGの表示用 state。** 「相性初到達で1回」を成立させるため、`GameSession` に `justPassedStage`（GameSession.swift:31）と同型の非RNGフラグ `didFireTsuukaChoice` を持たせ、発火済みを記録。**この state は runner の乱数を触らない＝golden 不変。** 専用アーク state が入ったら、アーク進行週との排他を同フラグ群で1コミット調停（→ 発火条件・境界）。

**(4) RNG/golden への影響＝確定トリガなら"なし"（0010 との差の核）。** A/B とも確定効果の純加算で、**発火も"帯初到達で確定"にすれば新規ロールを1回も引かない**＝乱数消費順は不変＝golden（3年ビット一致）不変。**【止めて相談】** もし将来この枠を「週次12%ランダム抽選」に載せ替えるなら、runner に新規ロールを差し込む＝**消費順が動いて golden が壊れ得る**（開発規律 A-1/A-2）。その場合の取り込み手順: ①`tools/*.py`（正典）に抽選位置/重みを先に定義 → ②GameCore 反映 → ③`gen_golden.py` 再生成 → ④`swift test` green を**1コミットで揃える**。**MVP第一弾は確定トリガで出し、ランダム化は相談後**（温度事故なしで先に入る・0010 型の段階投入）。

**(5) 会話プールと話者切替。** `Advice.name`（DialogueData.swift:11）に話者名の口は既にあるが、**イベント後会話を出す配線は無い**（現状 `innerVoice` は俺の独白のみ・谷口/相方は「将来レアイベントで」とコメント済）。本イベントが**その配線の初号**。A/B の後会話は相方 ID で分岐（谷口＝関西弁プール／他相方＝標準語プール・partner_id 実装後）。MVP は谷口プールのみ。**UI は swift test で検証不可＝simulator で相性を 15 まで上げて発火を目視**（開発規律 D-10）。

## リスク・注意

- **【前提是正・最重要】イベント基盤が未実装。** 「既存の12%抽選枠に載るだけ」は誤り＝抽選枠も相性帯フックも二択画面もコードに無い。本イベントは基盤の最初の建設対象。工数見積り・レビュー範囲をそれ前提にすること。
- **【止めて相談・条件付き】ランダム抽選化は golden。** 確定トリガ（帯初到達1回）なら新規 RNG 非消費＝golden 不変。週次12%抽選に載せ替える設計を採る場合のみ、消費順が動くので相談前に実装しない（(4)）。
- **【設計上の懸念】専用アーク・partner_id が未実装。** 排他の参照口も相方切替も無い。当面は谷口単独・確定トリガで出し、アーク実装時に発火週を調停、谷口以外は partner_id 実装後（本編）。谷口以外の標準語バリエーションは**MVP では死に枝**（発火し得る相方が谷口しかいない）＝本編用スキャフォールドと明記。
- **A の相性 −1 の理不尽感**。「崩す」で相性が下がるのは狙い（一度ほどく）だが"損した"だけに映らぬよう**センス +2 の見返りを明示**し、event_design 原則4（救い/回収の余地）を守る。sim で「A 後に相性を戻す導線（稽古/相方と過ごす）があるか」を EVENTS_ON 実測してから確定（体感を数値で直さない・ui_design §7-B）。効果は EVENT_FIRE_CAP=15 の総量予算に1回分計上。
- **明白な最適解を作らない**（event_design §2）。A（センス↑/相性↓/体力↓）と B（表現↑/相性↑/伸びしろ不動）が別軸のビルドに効くよう、大会審査の重みと噛み合わせて非対称を保つ。片方が常に上位互換なら失格。
- **口調ブレ**。俺＝全行標準語・計算屋・自己憐憫なし。谷口＝関西弁一貫（口癖「なんとかなるやろ」はここでは温存＝乱発しない）。谷口以外に関西弁を混ぜない。ボケ（オチ/ネタ本文）ゼロ＝全て観測と判断。既存作品（火花/べしゃり暮らし/M-1 実在ネタ）とのセリフ一致は各行照合し無し。固有名は架空・数値は【仮】。
- **デッドパンの一滴は"書いた俺と、読める谷口。それで足りている"の1個だけ**（5-6回に1回の規律）。後会話に二個目のウィットを盛らない（1テキスト＝技法1つ）。旧稿の地の文/俺セリフのオチ重複は解消済。
- **再発させない**（帯初到達1回）。ツーカー帯の"予定調和"を語るイベント自体が反復すると芯が崩れる。

## レッドチーム（A/B）

**A. 実装妥当性**
