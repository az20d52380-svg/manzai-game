3. **共有ブランチ `claude/manzai-slg-foundation-wprqlu` に触れない**（4.8が実装中）。
4. **成果物＝markdownの設計/決定/執筆doc**。自分のscratchpadに `NN_タイトル.md`（連番）で保存し、`INDEX.md` に「NN｜タイトル｜状態｜1行結論」を1行追記していく（進捗が見えるように）。**4.8が回収して実装・コミットする**＝あなたは書いて置くだけ。

## 最初の1本（必須・以降すべての土台）＝「現状→リリース」ギャップ棚卸し

- (A) 1年版デモの配布（事前登録/フィードバック用）／(B) 本編（多年キャリア）まで作り切る／(C) ストア審査に出せる最小。
- ※監査§1-3の指摘＝1年版は決勝に構造的に届かず全ラン負けで終わる。これは(A)なら"味"、(B)なら"未完"。**どれを前提に以降を書くか**を、想定と推奨つきで明示（オーナー判断待ちとして残しつつ、推奨前提で先へ進む）。

## 各docの型
- 冒頭に3タグ：`[状態=設計完 / 要オーナー決 / 要実装]` `[release=必須 / あると良い / 後]` `[golden影響=有 / 無]`。
- 本文：**結論 → 根拠 → 実装可能な仕様（or 本作トーンの本文）→ 残論点**。忖度なし・過剰は過剰と言う。
- ゲーム内文言は「」括り・本作トーン・数値【仮】・固有名架空。声/講評/イベント文は該当Skillの○×採点ゲートを通してから確定。

## 止まる/合図する時
- コードを書きたくなった → 書かず**仕様doc**に。
- golden/数値を確定したくなった → **推奨【仮】＋反証条件**で止め「規律A＝4.8管轄」と書く。
- オーナーしか決められない（リリース定義・業態等）→ **推奨を置いて両にらみで前進**（止まらない）。
- Skill採点で× → 直してから確定。
