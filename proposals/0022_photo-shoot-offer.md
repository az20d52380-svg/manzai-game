# 提案 v0 — 「撮られる仕事」若手特集の撮影オファー（華／知名度 hook・知名度20-50帯）

- 日付: 2026-07-06 ／ 数値は全て【仮】 ／ 既存ファイルは読むだけ ／ トーン＝ドラマ主軸（proposals/0009 確定）
- 対象読者: イベント／オファー機構実装担当・DialogueData / WeekMainView の書き手
- 位置づけ: 華・知名度 hook の単発。**華の"口"を増やす補充**（下記の前提訂正を参照）。見た目・露出（華／知名度）と芸の中身（稽古）が**同じ一週**を奪い合い、明白な最適解を作らない選択。

> **【レッドチームで前提を訂正】** 下書きは「6軸のうち華だけが動かす口を欠く死に軸」と書いていたが、これは**実コードと矛盾するため撤回**。`GameConfig.swift:134` の稽古 `.フリーライブ` は **華+3／表現+1／知名度(fame)+1／体力-30** を持ち、華を直接伸ばす。よって華は「口が無い」のではなく「**能動的な口が フリーライブ 1種＋自動2枠(W3/W6) に偏り、しかも フリーライブ は体力-30 の重い場数枠**」というのが正しいgap。本イベントは「**華を取る二つ目の口を、稽古と別の場面（撮影オファー）に、知名度帯ゲートで開ける**」補充と位置づけを改める。

## 狙い

華の能動的な口は現状 稽古 `.フリーライブ`（華+3・ただし体力-30）1種に偏る。フリーライブは体力を大きく削るため、体力管理と華育成が固く結びつき、**「華を取りたいが体力を割けない」局面で華を伸ばす代替が無い**。ここに、**体力を主コストにしない華の口（撮影＝顔と佇まいで華を取る）**を、状態が世界に映る形（知名度が中途半端に付いた若手にだけ来る撮影）で一枚差す。

要点は三つ。

1. **正解のないトレードオフ**（event_design §2・設計原則3）。A＝**華＋知名度＋小金**を取る代わりに**その一週は稽古を打てない**（機会費用）。B＝**稽古に回して中身(センス/発想)を確定で詰める**代わりに**華も露出も伸びない**。華+2 はフリーライブ華+3 より軽い体力コストで買えるがセンス/発想の確定+1 と拮抗させ、上位互換を作らない。
2. **帯ゲートで供給過多を防ぐ**。知名度(fame) 20-50 限定＋同キャリア原則1回。**下限20／上限50 は `GameConfig.swift:155` の `offerRates` が実際にfameを刻む閾（below20=5%・20-50=15%・50-80=30%）と一致**＝「オファーが増え始める中途半端な若手」という世界設定と機構が噛む（設計原則2）。
3. **相方依存の声で「顔で呼ばれる」の座り心地の悪さを出す**（0009）。芸で売りたい二人に、芸をしない一日の値札が付く。谷口＝関西弁、谷口以外＝標準語（声は相方IDで差し替え）。

## 実装イメージ（既存機構に載る形・コードは"提案"・golden影響は要相談）

**(1) 【MVP・推奨】既存オファー機構への「華/知名度特化バリアント」として載せる。** イベント選択機構は**未実装**（週次12%判定も選択肢イベントもコードに無い＝event_design_v0 の設計止まり）。`WeekMainView` は 1タップ=1週で、オファーは `offer: OfferSpec?` を週頭に提示する既存の行動枠。**華/知名度に振った `OfferSpec` を追加し、`20 <= state.fame && state.fame <= 50` の週にこのバリアントを提示する**のが、framework無しで今日動く最小形。A=このオファーを受ける（＝その週の行動）／B=提示を見送って稽古カードを選ぶ、が自然に「同じ一週の奪い合い」になる。

```swift
// 【提案・未適用】撮影オファー(華/知名度特化)。既存 OfferSpec 拡張案。
// GameConfig.swift:152-153 の offerMoney/offerExp と同じ枠。効果は確定加算のみ・RNG非消費。
// 提示条件: 20 <= state.fame <= 50（offerRates の 20/50 閾と一致・GameConfig.swift:155）。
public var offerPhoto = OfferSpec(
    name: "撮られる仕事",    // 若手特集の撮影
    income: 10_000,          // 【仮】小金
    fame: 1,                 // 【仮】知名度+1
    ability: (.華, 2),       // 【仮】華+2（フリーライブの華+3より軽い体力コストで買う代替口）
    stamina: 0               // 【仮】撮影は体力を主コストにしない＝フリーライブ(体力-30)との差別化
)
// B(断って稽古)は本バリアントを提示しない/選ばないだけ＝通常の稽古週。
```

**(2) 「その週は稽古を打てない」＝機会費用の持ち方。** 1タップ=1週の現行モデルでは、**撮影オファーを選んだ時点でその週の行動が埋まる**（バイト・休むと同様）。**特別な"稽古枠消費"機構は不要**＝下書きの「稽古アクションを1回消費」という専用スロット表現はレッドチームで撤回し、「その週の行動が撮影になる（＝稽古はできない）」に一本化。GameEngine のオファー適用（`GameEngine.swift:99` 近辺・知名度/能力/所持金の確定加算）に本 spec を通すだけ。

**(3) golden 非干渉の要。** 効果は華/知名度/所持金/(A時)能力の**確定加算のみ・RNG 非消費**。オファー提示は既存の週次判定の内側で、**イベント本体で新しい乱数を引かない**。この形なら**乱数消費順は変わらず golden（3年ビット一致）は不変**（proposal 0010 の「確定効果は golden 不変」原則を A/B 両方で満たす）。

**(4) 【止めて相談】B の「センス or 発想」を乱数で選ぶ場合のみ RNG に触れる。** どちらを+1するかを**決定的（現在値が低い方＝弱点補強等）**にすれば新規ロール無し＝golden 不変。**「ランダムでどちらか」にするとサブロールが1回増える＝乱数消費順が変わり golden が壊れ得る**ので実装前に必ず止めて相談（開発規律A-1/A-2）。取り込む場合の正典同期: ①`tools/*.py`（正典）に軸選択ロールの消費順を定義 → ②GameCore 反映 → ③`gen_golden.py` 再生成 → ④`swift test` green を**1コミットで揃える**。**推奨は決定的版**。

**(5) 会話プールと反復回避・話者切替。** `Advice.name`（`DialogueData.swift:10`）は既に「将来のレアイベントで谷口等に切替」を想定した口を持つ＝相方セリフは name 切替で載せられる。A/B の後会話は相方IDで分岐（谷口＝関西弁プール／他相方＝標準語プール）。`DialogueData.reactions` に `offer` 反応（現状「受けておくか。金は要る。」）とは別に本オファー用の反応行を足す。再発可にする場合は proposal 0001 の直近回避で固定文反復を薄める。**UI は swift test で検証不可＝simulator で fame 20-50 帯まで送って目視**（開発規律D-10）。

**(6) 【本編/後段用】popup型2択イベントとして出す場合。** event_design §2 の選択肢イベント機構（週次12%プール＋2択UI）が**実装されてから**。その framework はオーナー判断待ち（未実装）＝本節は framework 実装後の載せ替え先として置く。効果・会話は(1)-(5)と同一のまま流用できる。
