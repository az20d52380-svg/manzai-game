# 審査講評を実弱点(weakAbility)に配線する — 紙講評の"相対評価"化 提案 v0

- 日付: 2026-07-05（レッドチームA/B反映: 2026-07-06）
- 位置づけ: `judge_comments_v1.md` §1（紙講評・10年で最頻出コンテンツ）と `writing_craft_and_appeal_v0` §4-6「弱点を言語化し翌週の選択が変わる痛みと前進のループ」を**因果として配線する**提案。`is_proposal_only`＝既存ファイルは読むだけ、改変差分ではない。数値は全て【仮】、固有名は架空。

## 狙い

紙講評（`JudgeData.review`）の敗退フックは今、**体力/メンタル/相性/回戦帯**でしか分岐せず、プレイヤーの実際の最弱能力（`WeekMainView.weakAbility()` が既に算出している センス/発想/表現/華 の最小軸）に**紐付いていない**。結果、「ボケの手数不足」「ツッコミが全部同じ角度」等の一般論が、実状態と無関係に引かれ得る。

全国区の賞レース級の講評で刺さるのは「一般論」ではなく**そのコンビ固有の武器・型を基準に、今日はそこが崩れた**を言う"相対評価"だ（出典後述）。紙講評フックに `weakAbility` 次元を**優先レイヤー**として1枚重ねるだけで、講評は自動的に「このコンビの一番弱いところ」を名指し、`writing_craft` が繰り返す**講評→稽古の学習ループ**が初めて閉じる。決定的にはしない（`event_design_v0` §2「正解のない設計」を侵さない）。翌週の接続は"推奨"ではなく**淡い陰り**に留める。

## 対象箇所

- `ManzaiGame/Sources/JudgeData.swift` の `review(passed:state:salt:)` 敗退側プールと優先ラダー（現状: `stamina<30 → メンタル<40 → 一般fail`）。呼び出しは `TournamentResultView.swift` の1箇所（`salt = summary.week`・乱数非消費）で確認済み
- `docs/judge_comments_v1.md` §1「予選の紙講評」の増産表（回戦帯×通過/敗退）に **weakAbility 列**を追加する想定
- `ManzaiGame/Sources/DialogueData.swift` の `innerVoice(...)` — **講評を受けた翌週の"陰り"モノローグ**を増設。ただし発火には**新しい signal（前週の敗退講評が名指した軸）を GameSession から配線する必要がある**（後述リスク・"1段挿すだけ"では済まない）
- 名寄せ前提: `weakAbility()` が返す文字列 = `"センス" / "発想" / "表現" / "華" / "メンタル"`（相性は除外）。**注意: `weakAbility()` は メンタル も min の対象に含む**（`WeekMainView.swift` L403）。よって講評プールは craft4軸（センス/発想/表現/華）、メンタルは既存 `failMentalLow` が担う。翌週モノローグ側はメンタルの verdict も起こり得るのでキーを揃える（後述）

## どこから学んだか(技術＋出典URL)

- **相対評価＝当人の武器を基準に崩れを指摘する型**（本提案の核）: そのコンビ固有の型を基準に「今日は出た/崩れた」を言う講評が刺さり、定型の一般論は摩耗する — https://www.oricon.co.jp/news/2360524/full/ ／ https://note.com/shinypinkchina/n/nb2e8f5c9128a
- **ツッコミは"同じ角度"が構造的に飽きる／角度は分解できる**: ツッコミは①形式マーカー（テンポ維持）②意味フィニッシャー（可笑しさを補い笑いへ押す）に大別。単層だと単調 → 「角度」講評は**表現軸が最弱の時だけ**引く — https://www.jstage.jst.go.jp/article/wii/2/0/2_15/_article/-char/ja/ ／ http://www.tufs.ac.jp/ts/personal/kazama/shigen/1/Masuya.pdf ／ https://www.osaka-kyoiku.ac.jp/~kokugo/nonami/2020soturon/futiwaki.pdf
- **弱点は"本文の質"でなく構造層（手数/角度/間/設定/華）に型化できる**: 「わかりにくさ」は内容の質より接続不良・情報過多に宿る → weakAbility を構造層にマップすれば"ボケを書かない一線"を守ったまま名指せる — https://note.com/chekacheka/n/na1318648389e
- **"間/緩急"は独立した弱点次元**: テンポは同期と変調（遅→速の受け渡し）で成立。揃いすぎは弱点 → センス軸専用の"間"文面 — https://conference.wdc-jp.com/jass/43/contents/common/doc/P-22.pdf ／ https://vision.kuee.kyoto-u.ac.jp/japanese/happyou/pdf/Kawashima_2007_J_54.pdf
- **一筋の褒めは"構造の型"に照らす（Rule of Three / setup→payoff）**: 「三段目で外せた/二段で止まった」等、型基準の指摘は一般論に落ちない — http://kenlevine.blogspot.com/2017/07/the-comedy-rule-of-3s.html ／ https://nofilmschool.com/rule-of-the-three
- **弱点を"全部言わない"＝サブテキスト**: 事実1〜2個の観測に留め、原因・処方は言い切らない。痛みは"名指されたが処方は自分で選ぶ"余白から生まれる（谷口＝本音を言語化しない人物ルールと同型） — https://www.amazon.co.jp/dp/4845915839 ／ https://everplay.jp/column/33689
- **診断は"訂正"でなく"省察を促す問い"＝正解のない設計との両立解**: 即座の訂正より発問で自己省察を促す方が改善に効く → 講評は「コレを稽古しろ」と決定的に指示せず、翌週UIで"推奨されない稽古を淡く陰らせる"に留める — https://www.japan-sports.or.jp/Portals/0/data/ikusei/doc/curriculum/model_core_2014.pdf

## 実装イメージ(そのまま貼れるスニペット)

> **重要（B軸・表記規則）**: judge_comments_v1 §6-5 の**オーナー方針「笑」の字を不使用**は新設の審査員文言にも適用する（SNS想起回避）。以下の講評プールは「ウケ／沸く／届く」で代替し「笑」を使わない。
> **重要（A軸・Swift文字列）**: セリフ内の引用符は必ず `「」` を使う。ASCII の `"` を Swift 文字列リテラル内にそのまま入れると文字列が途中で閉じ**コンパイルが壊れる**。

### (1) JudgeData に weakAbility 別の敗退プールを新設（既存の Swift 配列と同形・提案）

// キー = weakAbility() が返す文字列（craft4軸のみ）。メンタルは既存 failMentalLow が担うので除外。
// 表記: 「笑」不使用（§6-5 オーナー方針）。引用符は「」で統一（ASCII " はビルドを壊す）。
static let failByWeak: [String: [String]] = [

    "表現": [

    "センス": [

    "華": [

### (2) 抽選の優先ラダー（提案・"弱点を捏造しない安全弁"付き）

```
review() 敗退側の優先度（提案・現状ラダーに1段だけ挿入）:
  1. stamina < 30            → failStaminaLow      （据置・急性の身体フックは最優先）
  2. メンタル < 40            → failMentalLow       （据置）
  3. 最弱craft軸の値が 2位と【仮】8pt 以上開く
       かつ 全軸が高水準でない → failByWeak[weakAbility]  ← 新設・優先レイヤー
  4. それ以外                → fail（既存の一般プール）  ← fallback
```

- 3の**閾値差ガード**が肝: 最弱が僅差 or 全軸が高水準なら「弱点が無いのに弱点を捏造」しないよう④の一般プールへ降ろす。呼び出しは `TournamentResultView` の1箇所なので、`review(passed:state:salt:)` は**シグネチャ据置のまま `state` の craft4軸から最小・2位を `review` 内で算出**すれば足りる（`weakAbility()` は `WeekMainView` の `private` なので外から呼ばない・同じ min ロジックを review 側に置く／または共有ヘルパへ切り出す）。**craft4軸の min はメンタルを含めない**点だけ `weakAbility()` と挙動が違うので、共有するなら「メンタル除外版」を明示する。
- 閾値 8pt と「全軸が高水準」の基準は**表示テキスト選択のための UI 定数【仮】**。数式・シミュレーションのバランス値ではないので `GameConfig` 必須ではないが、マジックナンバーにせず `JudgeData` 内の名前付き定数にする。
- 回戦帯別（`judge_comments_v1` §1のバンド）に振り分けたい場合は `failByWeak["表現_準決勝"]` のようにキーを帯付きにする拡張余地を残す。MVPは帯なしフラットで足りる。
- 通過側には weakAbility プールを足さない（痛みループは敗退駆動・スコープ限定）。

### (4) DialogueData の"翌週フック"＝淡い陰り（決定的にしない・§7省察の余白）

**このフックは「引数に既に weakAbility が来ているから1段挿すだけ」では動かない。** 現状 `innerVoice(...)` に来る `weakAbility` は**その週の live な最弱軸**であって、「前週の敗退講評が名指した軸」ではない。翌週になるまでに稽古で軸が入れ替わり得るので、講評が「発想」と言ったのに翌週の陰りが「表現」を睨む、という食い違いが起きる。よって：

- **GameSession に新フラグ `lastVerdictWeak: String?` を追加**（提案）。`pump()` で**敗退の stage 結果が確定した瞬間**に、その時点の craft4軸 min（review が名指したのと同じ軸）を格納。`justPassedStage` と同様に**行動を選んだら失効**（＝翌週頭の1画面だけ出る）。通過時・非stage週は nil。
- `innerVoice(...)` に `afterVerdictWeak: String?` を追加で渡し、**非nilかつ dict にキーがある時だけ**発火。優先度は `losing`（連敗2以上）の下・平常の上。メンタル敗退も起こるのでキーにメンタルも用意（無ければ発火せず平常へフォールバック）。

```swift
// 【提案・DialogueData.swift】前週の敗退講評が名指した弱点を受けた、翌週頭の内心。
// 発火条件: afterVerdictWeak != nil && キーが存在。優先度は losing の下・平常の上。
// "推奨"ではなく"指が止まる"程度の陰りに留め、別能力を伸ばす自由を残す。
// 語り部＝俺（標準語）。ボケない・過小申告と温度差の実況が基調。引用は「」。
private static let afterVerdictWeak: [String: [String]] = [
    "発想": ["審査員の紙、遠回しだったけど要は一本槍だ。まあ、槍が折れてないだけマシと思っておく。稽古表の「ネタ作り」のところで、少し指が止まる。やるかは、まだ決めてない。"],
    "表現": ["「同じ高さから降ってくる」か。褒められた気もするが、七割は刺さっていた。悔しいので稽古表を睨む。"],
    "センス": ["「崩す度胸」ね。言うのは簡単だ。……とか言いつつ、「ネタ合わせ」の欄を二回見た。"],
    "華":   ["前に出ろ、と紙に書いてあった。俺が一番、後ろに立ちたがる質なんだけどな。"],
    "メンタル": ["紙は優しかった。優しさが逆に効く時があると、初めて知った。今週はまず、寝る。"],
]
```

## リスク・注意

- **翌週フックの配線コスト（A軸・最重要の誤解訂正）**: 上記(4)の通り「引数に既に weakAbility が来ている」だけでは食い違う。`GameSession.lastVerdictWeak`（新フラグ・行動で失効）＋`innerVoice` の引数追加＋`WeekMainView` 呼び出し側の1行改修が要る。MVP に save/load は未実装なのでセーブ整合は問題にならないが、将来シリアライズ対象を作る時は本フラグも含める。
- **決定的にしすぎる罠（§2）**: 講評→稽古を"推奨"として明示配線すると「正解のある稽古ゲー」に堕ち、`event_design_v0` §2 と衝突する。翌週の接続は**淡い陰り止まり**にし、別軸を伸ばしても罰しない。UIで稽古カードをハイライトするなら"淡い"を厳守。
- **弱点の捏造**: 閾値差ガード（【仮】8pt・全軸高水準は除外）を外すと、強いコンビにまで弱点を名指してしまい講評が嘘になる。ガードは必須。
- **名寄せの一致と メンタルの扱い**: `weakAbility()` は メンタル も min 対象に含む（`WeekMainView.swift` L403）。`failByWeak` は craft4軸のみ（メンタルは `failMentalLow`）。1文字でもキーがズレると `nil`→一般プール沈黙。**単一の定数配列（またはenum）で軸名を共有**するのが安全。翌週フックはメンタルキーも持たせて食い違いを塞ぐ。
- **状態フックとの二重発火**: メンタル<40 は既存 `failMentalLow` が担う設計を維持。stamina/メンタルの急性フックは weakAbility より**優先**に置く（身体の崩れが今日の直接因なので）。
- **口癖・型の乱用（writing_craft/天丼ルール）**: 「〜ということで」等の締めや「事件」「崩す勇気」「武器」等の既存頻出語は、プールを跨いで反復すると摩耗する。本提案では既存 §2 と重なる語を意図的に一段ずらした（下のレッドチーム節参照）。谷口の口癖「腹減ったな」は本提案では使わない（`dialogue_design_v0` §年2回制限を尊重）。
- **golden/数式・乱数消費順**: `review()` は乱数非消費（salt=week の決定的インデックス）・UI層データ。**golden にも数式にも触れない**ので `swift test` 正典同期の対象外。もし将来この判定を GameCore 側へ動かす・抽選に rng を使う設計に変えるなら、その時は**手を止めて相談**（消費順が golden を壊す）。

## MVP範囲

- 実装は `review()` 内の min 算出＋プール1本（シグネチャ据置）と、翌週フックの `GameSession.lastVerdictWeak` 配線に収まる。回戦帯別キーは拡張余地として置くだけ。

## レッドチーム（A/B）

本節は起草後の自己攻撃と対処の記録（2026-07-06）。実コード（`JudgeData.swift`／`DialogueData.swift`／`GameSession.swift`／`WeekMainView.swift`／`TournamentResultView.swift`）を読んで確認。

### A. 実装妥当性
