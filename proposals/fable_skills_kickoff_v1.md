# Claude Code Skill 作成キックオフ ―― モデル頑健な「声」のSkill化

**ultrathink（ウルトラシンク）。書き始める前に深く考えろ。** これは文章を書く依頼ではなく、**「文章の品質を"手順"に固定して、別のモデル（Sonnet や Opus 4.8）が書いても品質が落ちないようにする」設計**の依頼だ。あなたが持っている"寒くしない勘"を、勘のないモデルでも踏める**チェックと手順**に翻訳することがゴール。まず§0〜§3を読み、何を手順化すべきかを設計してから着手しろ。

## 0. ゴールと、なぜ書き方が変わるか

- ゴール: この作品の「声」を**Claude Code の Skill（`SKILL.md`）**にする。Skill を読めば、**Sonnet でも Opus 4.8 でも、あなたと同じ水準の地の文・会話が書ける／既存の原稿を同じ基準で採点・修正できる**状態を作る。
- **なぜ普通の文章と作り方が違うか**: 弱いモデルほど「なんとなく寒い」を検知できない。だから Skill は**"良い趣味を持て"と言ってはいけない**。代わりに――
  - 感情語を物へ変える**操作手順**（レシピ）、
  - 書いた後に必ず走らせる**○×自己採点＋×が出たら直す手筋**、
  - **良い例と悪い例を"なぜ寒いか"付きで対にした見本**、
  - 迷いを消す**判定表**（どの物を選ぶ・どちらの話者・ユーモアの種類）、
  - **ハード上限**（ユーモアは5-6本に1本・物は1個・話者ごと1セリフ）、
  ――を載せる。**判断を全部"手順"に落とす**ほど、モデルが変わっても品質が落ちない。これが唯一の設計指針だ。

## 1. Claude Code の Skill とは（正確な仕様）

Skill は「特定の仕事のやり方をモデルに教える、フォルダ＋Markdown」。仕様は厳密に守れ。

- **置き場所**: `.claude/skills/<skill-name>/SKILL.md`（プロジェクト共有）。`<skill-name>` は小文字・ハイフン（例 `manzai-drama-voice`）でフォルダ名と一致。
- **SKILL.md の先頭に YAML フロントマター（必須2キー）**:
  ```
  ---
  name: manzai-drama-voice
  description: <三人称で「何をするSkillか＋いつ使うか」。トリガー語を具体的に。最大約1024字。>
  ---
  ```
  - **description が最重要**＝モデルはこの一文だけを見て「このSkillを使うか」を判断する。抽象語でなく**発動条件のキーワード**を入れる（例: 「漫才師育成SLGの日本語の地の文・会話（心の声／相方会話／審査講評／選択肢イベントの会話）を新規に書く・または既存原稿を採点/修正するとき。ドラマ主軸の声を全モデルで一定に保つ」）。
- **本体（フロントマター以降）**: Markdownの指示。**簡潔に（目安500行以内）**。詳細は別ファイルに切り出し、SKILL.md からは「必要な時に references/xxx.md を読め」と**参照で繋ぐ**（＝プログレッシブ・ディスクロージャ。全部を一度に読ませない）。
- **付随ファイル**: 同フォルダに `references/`（規約・見本・チェックリスト）等を置き、SKILL.md から相対パスで指す。モデルは必要になった時だけ開く。
- **任意**: `allowed-tools:` をフロントマターに書けば使用ツールを制限できる（この用途では基本不要）。
- **テストの考え方**: Skill は「description で正しく発動」「本体の手順が曖昧さゼロ」で価値が決まる。**曖昧さ＝弱いモデルでの品質低下**。

## 2. 作るSkill（1つを主・2つは任意）

### 主（必ず作る）: `manzai-drama-voice`
この作品の地の文・会話を**書く／採点する**ための中核Skill。ファイル構成:
```
.claude/skills/manzai-drama-voice/
  SKILL.md                     … 簡潔な発動条件＋手順の骨格＋"必ず走らせる採点ゲート"＋references案内
  references/voice-bible.md     … 声のロック・芯・アンチ寒さ則・固有/定番の原理（規約の正本）
  references/examples.md        … スロット別の【良い例／悪い例（なぜ寒いか付き）】の対
  references/slots.md           … スロット地図＋固定スロット名⇄コードキー＋話者
  references/self-check.md      … ○×採点表＋×が出たときの"直す手筋"（修復レシピ）
```

### 任意（主が仕上がってから・オーナー判断）
- `manzai-choice-events` … パワプロ型の選択肢イベントを設計するSkill（正解のないトレードオフ・効果は"方向"だけ言葉に噛ませ数値は書かない・ボケ禁止・golden安全＝乱数を引く効果は"止めて相談"）。
- `manzai-judge-comments` … 審査講評を書くSkill（7審査員の人格・辛口は必ず「直せる一点」・「笑」不使用・実在審査員模倣NG）。

## 3. モデル頑健化の技法（この設計の心臓・全Skill共通）

弱いモデルでも品質を落とさないため、各Skillに**必ず**次を埋め込め:

1. **前口上（restate gate）**: 書く前に「基調（ドラマ主軸・アンチ寒さ則・声のロック）を自分の言葉で3行に要約せよ」と手順で強制。ズレたまま書き出す事故を止める。
2. **感情→物の変換レシピ（機械手順）**: 「①感情語を一度書く→必ず消す→②その瞬間、手/目/物が何をしていたかを1つだけ書く→③因果を説明せず事実を並置→④ピーク直前で『〜た。』で止める」。趣味でなく操作として書く。
3. **○×自己採点ゲート＋修復手筋**: 1本書くごとに走らせる10項目前後の○×（感情語なし／物は置換不能か／俺にキザ・自己憐憫が漏れてないか／ユーモアは5-6本に1本か／話者ごと1セリフか 等）。**×が出たら"どう直すか"の手筋**（物を1個に減らす・警句を消す・次の文を平常の事実に戻す）まで書く。採点だけで終わらせない。
4. **良い例／悪い例の対（なぜ寒いか付き）**: スロットごとに、**悪い例に「これは感情語の言い換え」「これは計算屋のキザ」等の理由タグ**を付ける。弱いモデルは"なぜダメか"が言語化されて初めて避けられる。
5. **判定表で迷いを消す**: 「どの固有物を選ぶ？→他コンビで成立しない履歴/固有名詞に紐づく物、置換可能なら没」「谷口=関西弁／他相方=標準語／第三者=標準語」「ユーモアは自嘲orデッドパン、各分岐最大1個」を表で断定。
6. **ハード上限を数で明記**: ユーモアの分母（連続で引く並びの5-6本に1本＝各スロット最大1本）／固有物1個／会話は話者ごと1セリフ（「／」で詰めない）。曖昧な"控えめに"を数に変える。
7. **少数精鋭＋停止規則**: 「浅い量産が最大の失敗。天井であってノルマではない。寒い/キザ/自己憐憫が出たら即引く」。
8. **自己テスト指示（Skill内に）**: 「この手順を"勘のないモデルのつもり"で1スロット試し、○×が全部○になるまで直してから残りへ」。

## 4. 声の規約（自己完結・references/voice-bible.md の中身）

（Skillは本を読めない前提。較正に要る規約は全て内蔵する。詳細な正典は `proposals/fable_kickoff_drama_allin_v1.md` にあるが、Skillはそれ無しで成立させる。）

- **芯**: ゲームは**ボケ（オチ/フリ/ネタ本文/パンチライン）を一文字も書かない**。笑いは地の文（心の声/場面描写/審査講評）にのみ宿る。例外＝審査講評が「オチ/ボケ/テンポ」を批評語として使うのは可。
- **基調＝ドラマ主軸**: 感情語（悲しい/悔しい/惨め/嬉しい）を書かず、感情が起きた**「場所の物」**に背負わせる。
- **ユーモアは二本立ての乾いた一滴のみ**: 自嘲 or デッドパン（真顔で事実の並置・自己言及ゼロ）。5-6本に1本だけ。
- **固有/定番の原理**: 固有＝その稽古場/相方/客席の固有名詞・履歴に紐づき他コンビで成立しない物。定番（天井の染み・冷めたコーヒー・小銭・家賃・財布）は**置換可能＝没**。物は1個。
- **声のロック**: 俺＝標準語・計算屋・ツッコミ体質だが**自己憐憫もキザ（うまいこと言う警句化）も禁物**。熱・執念は俺でなく谷口・支配人へ寄せる。谷口＝関西弁で本音は言語化しない。谷口以外の相方・第三者＝標準語（関西弁を移さない）。
- **会話**: 話者ごとに1セリフずつ（「／」で詰めない）。壁は回戦名＋経過年で名指す。
- **恒久規約**: 固有名は架空・数値は【仮】・既存作品（火花/べしゃり暮らし/M-1実在ネタ）のコピペ酷似厳禁・「笑」の字不使用・実在審査員模倣NG。
- **スロット**（references/slots.md）: 心の声（default/lowStamina/lowMoney/losing/passedLines/preTournament/reaction）／審査講評（verdict.pass/fail・final.B/D）／山場（決勝敗退・優勝・年次独白）／相方会話／選択肢イベント（event:採番.setup/.A/.B/.C）。

## 5. 出力（何を・どの形で返すか）

- **各Skillごとに、フォルダ内の全ファイルを"ファイルパス＋中身"の形で**出せ。例:
  ````
  ### .claude/skills/manzai-drama-voice/SKILL.md
  ```markdown
  ---
  name: manzai-drama-voice
  description: ...
  ---
  （本体）
  ```
  ### .claude/skills/manzai-drama-voice/references/voice-bible.md
  ```markdown
  （中身）
  ```
  ````
- **SKILL.md 本体は簡潔に**（発動条件の再掲＋手順の骨格＋"必ず走らせる採点ゲート"＋「詳細は references/ を読め」）。分量は references/ に逃がす。
- **description は具体トリガー語つき三人称**で。抽象語だけにしない。
- 全ファイルに**冒頭1行**で「これは種・数値は【仮】・最終確定は実機目視」を記す。

## 6. 進め方（Skill内の自己テストと同じ姿勢で）

1. **基調確認**: §4の規約を自分の言葉で3行に要約し、§3の頑健化技法をどう各ファイルへ割り付けるか設計する。
2. **主Skillを1つ完成**: `manzai-drama-voice` の5ファイルを作る。**references/self-check.md の○×を"勘のないモデルのつもり"で1スロット試走**し、全部○になるまで手順・見本・判定表を直す（曖昧さが残る＝弱いモデルで崩れる）。
3. **任意2つ**: 主が固まったら `manzai-choice-events`／`manzai-judge-comments` を同じ設計で。時間・分量が尽きたら**主Skillだけで止めてよい**（主が完成品であることを優先）。
4. **禁止事項**: 数式・バランス・golden・乱数消費順には触れない（Skillは文章＝UI層テキストの作法のみ）。効果数値を書かせるSkillにしない（選択肢イベントは"効果の方向"を言葉に噛ませるだけ）。

―― 以上。深く設計し、"勘"を"手順"に変えろ。始めてください。
