# 経験点ループ「稼ぐ→貯まる→注ぐ→伸びる」体験設計 v0

Fable回答（体験設計・2026-07-13）。コード納品なし・実装はMac側。本書のSwift断片は全て例示。数値は全て【仮】・固有名は架空。正本参照: `docs/exp_abilityup_impl_reply_v0.md`（機構v2）・`ui_redesign_pawapuro_uma_v0.md`（v8確定・追補⑥）。

**一行要約**: 稼ぐ瞬間は「+N能力」ピルを**粒チップ**（塗りドット=同色／輪郭ドット=共通——AllocationView・NotebookViewと同一文法）へ置き換え、受け取りは**入力を止めない1.7秒の一拍**だけにする。「のばす」バッジは粒総数から**「いま注げば伸びる段数」（`recommendedPlan(s).count`）**へ意味を変え、空振りを構造で消す。満了は既存の一文「この年の器は、満ちた。」を三つの面で共有し、以後カードは粒を語らず**金の「満」判**を置く。新規ハプティクスはゼロ、新規ゲーム内文言は5つだけ。

### 設計原則（3つ）
1. **姿が仕様を言う。** 塗り=行き先が決まっている、輪郭=まだ決まっていない。この2形だけで二区画を言い切り、「75%」「ロック」「共通枠」という語をUIに出さない。稼ぐカード→受け取り→帳面→注ぐ画面まで、粒チップは**同じ1部品**で通す（初見の教育コスト=1回）。
2. **稼ぎは一拍、注ぎは儀式。** 週送りの0.2s文法を1msも壊さない。受け取りは非ブロッキングの余韻だけ、祝い（リビール・emphSpring・confirm振動）はAllocationViewが既に持っている——稼ぎ側で二重に祝わない。**本設計は新規ハプティクスを1つも足さない。**
3. **責めず、隠さず。** バッジは「いま注げば伸びる段数」だけを数える（点いていれば必ず何かが伸びる、が不変則）。帳面（Notebook）は常に全真実。満了は同じ一文で三面が言い、以後は黙る。点滅・赤・モーダル・週越しの再掲はどれも使わない。

**却下史への忠誠（1行）**: 中間経験点プールは「どの稽古を選ぶか」の希薄化を理由に2度却下され、同色75%の主従で復活した経緯がある（ui_redesign追補④/⑥→機構v2）。だから稼ぎの絵でも**主従を崩さない**——同色チップが常に先頭、共通チップは末尾・従。稽古選択の重みはチップの並びが守る。

## 1. 稼ぐ瞬間 — カード予告と受け取りの一拍

### 1-1 粒チップ（1部品を全面で共有する）
「5能力色＋2グループの異なる粒度が混在」を1目で分からせる答えは、**新しい記号を作らないこと**。AllocationViewが既に確立した2形をカードサイズに縮めるだけ:
| 種 | 形 | 例 |
|---|---|---|
| 同色ロック粒 | **塗りドット**（`Theme.abilityColor(a)`・6pt【仮】）＋能力名＋数値。地=`card2`カプセル | `● 発想 +2` |
| 共通粒 | **輪郭ドット**（`inkDim` stroke 1.5pt）＋グループ名（`ExpGroup.rawValue`=「ネタ」「舞台」）＋数値。地=`card`カプセル＋`line`縁 | `◯ ネタ +1` |
- **数値の規約**: カード予告と受け取りは増分「+N」、帳面と注ぐ画面は残高の素の数。Nは**プール残高の丸め差分**（`Int(after)−Int(before)`）——v8⑧の整数表示規約を粒に延長し、**予告と着地が同じ関数・同じ丸めで必ず一致**する（見積もり=確定の単一化の家風）。
- **差分0のプールはチップ自体を出さない**（「+0」を印字しない）。端数繰り越しで同じ稽古でも◯ネタが出る週と出ない週があるが、「出た数字は必ず本物」の方が「毎週+0.75と書く」より誠実で静か。
- **色覚への二重符号**: 塗り/輪郭は色でなく**形**の差、かつ共通チップは必ず文字ラベル（ネタ/舞台）を伴う。色が読めなくても文法は読める。
```swift
// 例示: 粒チップ（AllocationView.lockedChip / freeChip をカード用に共通部品化するイメージ）
HStack(spacing: 3) {
    Circle().fill(Theme.abilityColor(a)).frame(width: 6, height: 6)   // 共通粒は .stroke(Theme.inkDim, lineWidth: 1.5)
    Text("発想 +2").font(.system(size: 9.5, weight: .bold)).foregroundStyle(Theme.ink)
}
.padding(.horizontal, 5).padding(.vertical, 2)
.background(Theme.card2, in: Capsule())
```

### 1-2 カード面 — `gainPills` 枠の置き換え
- **位置・行高は現行のまま**（WeekMainView.swift:336-340 の分岐の中身だけ差し替え）。稽古カードの伸び行=粒チップ行（max3・prefix既存）: **主同色 → 副同色 → 共通**の固定順。共通チップは常に末尾——75/25という数字を出さずに、並びと大きさが主従を言う。
- **直接効果は旧文法のまま残す**: 相性・体力・金など経済外の即時効果は現行の塗りカプセルピル/コスト行のまま。**「塗りピル=いま直接効く」「ドットチップ=貯まる粒」**——即時と貯蓄が形で分かれ、経済の内外がそのまま見た目の内外になる。
- 全プール差分0の稽古週（副収穫の端数のみ等・稀）: 現行「伸びわずか」の位置に**「粒わずか」**（inkFaint・同書体）【仮】。追補⑥（A/B/C・オーナー保留中）の裁定が出たら語を合わせて差し替え。
- 器満了後のカード面は§4（チップ行→「満」判）。

### 1-3 受け取りの一拍 — 週送りに1msも足さない
実行の既存文法（押下→引き抜き→tick→週送り）は不変。粒の受け取りは**週送り後の画面で、状態差分駆動**で起きる（大会・イベント画面が挟まっても、週メインに戻った時に成立する——演出の振り付けに依存させない）:
| t | 対象 | 動き | トークン |
|---|---|---|---|
| 0.00 | （既存）押下→引き抜き→`Haptics.tick()`→週送り | 不変 | 既存 |
| 週メイン表示直後 | 獲得チップ行（今週入った粒チップ2〜3個） | 「のばす」タイル直上に下から8pt浮き上がって出現 | appear=easeOut 0.25s |
| +0.10s | バッジ | 数字が繰り上がり＋タイルが一拍（scale 1.0→1.05→1.0） | `.numericText()`＋`emphSpring` |
| +1.30s | 獲得チップ行 | 消える | exit=easeIn 0.18s |
- 合計約1.7s・**入力は一切遮断しない**。次のタップで即消え（48週×周回で摩耗しない長さ。実機で長ければ滞留1.2→0.8s【仮】へ）。
- **ハプティクスは追加しない**（tickが既に鳴っている。受け取りはその余韻であって新しい事件ではない）。
- SEは未統合のまま成立させる。将来足すなら受け取りの1音のみ【仮】（紙に小豆が落ちる程度の短音・満了後は鳴らさない）。

### 1-4 初回だけの一言（吹き出し・各1回）
共通粒チップが**初めて数字を出した**受け取りの時だけ、チップ行の上に白吹き出し（e1・serif 12pt・2.5s）:
- ネタ初回: 「ネタの粒は、センスにも発想にも注げる。」
- 舞台初回: 「舞台の粒は、表現にも華にも注げる。」
読み飛ばされても損はない——本教育はAllocationViewの**枠の包含**（ネタ枠カードの中にセンス行と発想行・共通チップはヘッダ）が既にやる。これは予習の一言。

> **実装メモ**: 流用=`gainPills`の分岐枠・`freeChip`/`lockedChip`の意匠・`previewState`（粒差分は`exp*`のbefore/after——v2 §4-1の「lastGainsを粒差分に振り替え」指示そのもの）・トーストidiom・`.numericText()`。新規=粒チップ共通部品（private からの昇格・新ファイルなら`xcodegen generate`）・`GameSession`のlastGrainGains・初回フラグ2つ（UI層・UserDefaults【仮】）。

## 2. ループのペース — 「即時が既定・貯めは自由」を所作に写す
機構の正典（v2論点B）が既に答えを持っている: タイミングは決定として立てない・銀行は中立バッファ・貯めても機構上の得はない。体験側はこれを**そのまま**写す——どのリズムも罰しない・矯正しない。
- **想定される所作の推移（全部正解として設計）**: 序盤=毎週開けて確かめる（学習）→ 中盤=バッジが2〜3段になったら開ける（バッチの快感: 複数リビールが一度に走る）→ 本番前=必ず一度開けて「つぎの本番」カードの要求ラインに寄せる。
- **注いだ能力は次の本番から働く**——だから「気づいた時に注ぐ」が実利の最適で、それを支えるのがバッジ。逆に数週放置しても何も失わない（中立バッファ）から、**放置に対して何も起こさない**（再掲・点滅・エスカレーションなし）ことがバッチ派の遊びを守る。
- **「のばす」は週を消費しない**（kind .info・dataタイルと同型）。この安心は既存文法が既に教えている——新しい説明を足さない。
- **バッジの意味論変更（本書の核心の一つ）**: 現行`Int(expTotal)≥1`（粒総数）→ **`recommendedPlan(s).count`（いま注げば伸びる段数）**。見た目は現行のgainOrangeカプセル数字のまま、意味だけ変わる。効果: ①器満了・上限・逓減死で自動消灯（「開いたのに何もできない」空振りがゼロ）②数字が「行けば起きる良いこと」の個数になる（粒総数34より、のばせる7の方が誘いとして正直）。
- **本番前の一段強い合図**: 次の大会の**前週【仮】**かつバッジ≥1のとき、バッジに`verm`のリング（stroke 1.5pt）を足す。朱=「つぎの本番」カードの要求ラインと同じ語彙——「本番が呼んでいる」を色の意味の再利用で言う。トーストは出さない。実機で見落とし報告が出た時だけ、既存文言「注いでいない粒がある。「のばす」から注ぐ。」を大会前週頭の1回トーストに昇格する（**予備・新規文字列なし**）。
- **作業感ガード（v2論点Bの反証条件への備え）**: 「毎週開いて注ぐだけ」が作業として出たら、まず疑うのは画面でなく**開かせ方**——バッジを放置できる安心が壊れていないか（点滅・催促を誰かが足していないか）を先に点検する。畳む方向（週末自動おすすめ注ぎのオプション）は機構不変で足せるとv2が保証済み。

> **実装メモ**: 流用=バッジ実装（WeekMainView.swift:265-274・条件式と表示値の差し替えのみ）・`recommendedPlan`（既存純関数・RNG非消費・**表示専用なのでgolden非干渉**）・次大会ヘルパ（AllocationView.swift:186-191）。新規=バッジ値の週次キャッシュ（UI層）・vermリング条件1つ。

## 3. 未割り振りへの気づかせ — 四つの面・全部静的
「注ぎ忘れを責めず、見落とさせない」は、**能動的通知ゼロ・常設の面を4つ**で解く:
| 面 | 言うこと | いつ | 形 |
|---|---|---|---|
| 受け取りの一拍（§1-3） | 「入った・そこに行った」 | 稼いだ週だけ | チップ行＋バッジ繰り上がり |
| バッジ（常時） | 「いま注げば、この段数伸びる」 | のばせる時だけ点灯 | gainOrangeカプセル数字 |
| vermリング（§2） | 「本番が近い」 | 大会前週×バッジ≥1 | バッジの朱縁 |
| 帳面（Notebook growthRoom） | 残高の全真実（色別・共通別） | いつでも | 既存のまま |
- **禁じ手**: 点滅・赤・モーダル・週越しの再掲トースト・「まだ注いでいません」類の呼びかけ文。気づかせの主役は**受け取りの一拍が毎週バッジの位置を指差し続けること**——導線の学習は演出がやる、催促は誰もやらない。
- **満了時の整合（1条件の修正）**: growthRoomの誘導文「注いでいない粒がある。「のばす」から注ぐ。」は現行`hasGrain`のみで出る（NotebookView.swift:133）。器満了中は「注ぐ」が嘘の導きになるので、**表示条件に「注げる段があること」を足す**（バッジと同じ`recommendedPlan`非空・ソースを共有して食い違いを構造で消す）。満了中は器の一文（すぐ上に既存）だけが立つ。

> **実装メモ**: 流用=growthRoom一式（NotebookView.swift:91-138）。新規=誘導文の条件1つ（`hasGrain && !plan.isEmpty`）。文言の追加・変更なし。

## 4. 余剰と満了 — 「損」ではなく「今年の伸びは仕込み終わった」
**事実の整理（設計の土台）**: 器は注ぐ瞬間だけ消費する（稼ぎは予算非消費）→ **満了は必ずAllocationView内で起きる**。粒は年末に消えない（リセット処理なし・v2「ゼロ利得段は非消費」=余剰許容の実体）。MVPは1年周回なので、満了後に稼いだ粒は**この周回では注がれない**——「無駄になる」のは事実で、隠さない。ただし語るのは一度だけ、以後は絵が引き受ける。

**満了の一拍（初回のみ・AllocationView内・実装1点の追加）**:
| t | 対象 | 動き | トークン |
|---|---|---|---|
| リビール完了後 | 器の金バー | `gold`のきらめきが左端→右端へ一度だけ走る | 0.4s easeOut |
| +0.4s | 器の文言 | 「まだ、伸びしろがある。」→**「この年の器は、満ちた。」**へクロスフェード | std 0.25s |
追加ハプティクスなし（「注ぐ」の`confirm`が直前に鳴っている——満了はその結果であって新しい事件ではない）。

**週メインへ戻った初回（1回だけ）**: トースト1.4s（既存infra）: **「この年の器は、満ちた。」**——器カード・Notebook・トーストの**三面が同じ一文**を言う。新規文字列ゼロ。この統一が「どこで読んでも同じ状態」の安心を作る。

**以後の稼ぎ側（抑制表示——使えない通貨を祝わない）**:
- **カード**: 粒チップ行の位置に**「満」判**——`gold`縁の角判（`Theme.Rad.stamp`=4pt・`goldD`の一字・10.5pt【仮】）。合否押印（TournamentResultView）と同じ判の語彙で、「損した」でなく「押印済み=今年の伸びは仕込み終わった」と読ませる。コスト行・相性など経済外の直接効果ピルはそのまま残る——**稽古の他の意味は生きている**（体力と金の使いどころ・相性・大会だけの年後半という別の遊びへ、カードの絵が自然に切り替わる）。
- **受け取りの一拍**: 出さない。**バッジ**: 自動消灯（§2の帰結・追加実装なし）。
- **帳面**: 正直に増え続けてよい。器の一文がすぐ上で理由を言っており、「粒はあるが、この年はもう注がれない」は帳面として読める真実。
- **会計には触らない**（発行継続のまま・UIの抑制のみ）。発行停止という代替案は§6-2でMac側判断に委ねる——どちらでも本書は成立する。

**将来への含み（multi-year化した時に効く・MVPでは実装しない）**: 粒が消えない設計のおかげで、年替わりで器がひろがった瞬間に**バッジが再点灯**する——去年の余りが新年最初の贈り物になる（「捨てた」が「待っていた」に反転する）。文言候補【仮】: 年頭「二年目のぶん、器がひろがった。」（v1候補の流用）／growthRoom満了文の隣に「粒は、来年の器を待つ。」

> **実装メモ**: 流用=器金バー・トーストinfra・`Rad.stamp`・判の意匠。新規=きらめき1本（AllocationView・表示のみ）・「満」判・満了初回トーストのフラグ（UI層）。

## 6. Mac側への確認事項（実装前に1行ずつ決めたい）
1. **副収穫のρ扱い**: ネタ見せ会の副メンタル等、稽古の系とグループが一致しない副能力の25%はどこへ行くか。**推奨=メンタルは全量ロック**（「メンタルは同色のみ」の素直な読み。舞台枠へ混ぜるとメンタル稽古で舞台共通が太り、位置=意味の判読性が濁る）。
2. **満了後の発行**: 継続（本書の前提・UI抑制のみ・会計単純）か停止（帳面も止まる=より正直だが状態分岐が増える）か。どちらでも本書の§4は成立。
3. **バッジの`recommendedPlan`呼び出し**: 純関数・RNG非消費・表示専用でgolden非干渉のはず。ただし呼び出しは**UI層に置く**（GameCoreの週処理に入れない）——消費順の正典に指一本触れないため。
4. **「伸びわずか」追補⑥（A/B/C・オーナー保留中）**: 本書は独立に成立する。「粒わずか」の語だけ、裁定が出たら合わせて差し替え。
5. **vermリングの発火幅**: 前週のみ【仮】か2週前からか——実機の見落とし率を見てから。

**導入順の提案**（会計移設の後・各段が独立に出荷可能）: §1カード置換＋受け取りの一拍 → §2バッジ意味論 → §3誘導文の条件 → §4満了の抑制表示。§1が入った時点でループは一応閉じる。
