## 0. 結論（3行）

1. **ネタは「年始に1つ選ぶ型」ではなく、`ネタ作り` で1本ずつ生まれる個体**。`ネタ見せ会`/`フリーライブ` で客前にかけると**ウケ（客反応）で完成度が上がり**、**持ちネタ帳（レパートリー）に貯まって年を跨いで持続**する。何年も前のネタを今年の大会で選んでもよい。
2. これは**新機構ではなく、既存の稽古アクション（`GameConfig.swift:150-155` の5種）・大会/GP（`Calendar.swift:59-79`）・採点式（`GameEngine.swift:145-160`）・見せ札演出（`FinalsPresentationView.swift`）・ネタ帳UI（`NotebookView.swift`）を"ネタ個体"で束ね直すだけ**で成立する。GameState へのフィールド追加は `exp*` 追加（`GameState.swift:30-38`）と同型＝**Codable自動・golden不変**（`CodableTests.swift:11`）。
3. **Phase 0（golden安全＝本書の主役）＝ネタ個体を state に持ち・作り・磨き・帳簿に貯め・大会で"どれで出るか選ぶ"まで、合否スコアには一切効かせない**。7審査員×型の相性は**講評/見せ札tiltの演出材料（非スコア）**として残す。**"ネタが合否スコアに効く"のは Phase 1＝規律A（tools/*.py→GameCore→gen_golden→swift test を1コミット・sim較正必須）で明確に別掲。新しい乱数を引く設計に至ったら止めて「4.8相談（規律A）」と明記する**。

## 1. 実在漫才の要点（調査1〜5の要約・設計が背負う事実）

本設計が忠実再現すべき、実在漫才の**構造的事実**だけを抜く（出典は §9）。

| # | 事実 | 設計への要請 |
|---|---|---|
| A | ネタは**1本ずつ作る**。「テーマ→ボケ大量出し→(他人に見せて)選別→構成→台本化」で1〜2週間（調査1 §2） | `ネタ作り` は**1本の個体を生む/伸ばす**行為。抽象値+3では欠落 |
| B | **新ネタは"おろす"（初披露）→劇場・営業で場数→客反応で削り直す**。「漫才はドラクエ、やればやるほど経験値」（調査2 §2/§6） | **ライブでウケると完成度が上がる**。舞台数の反復が完成度を押し上げる |
| C | **客の反応でウケ/スベりが可視化**され、それが改稿の羅針盤（調査2 §3）。会場・客層で反応が変わる（場数=実験） | ライブの**ウケ（客反応）を個体に記録**し、磨きの手応え＝反応として見せる |
| D | **持ちネタは芸歴で貯まる資産**。**古いネタの再演が主流**（アンタッチャブル/タカトシ=20年超前を更新して現役）。良ネタの寿命=十数年〜20年超（調査3 §3/§4） | **ネタは年を跨いで持続**。今年作ったネタしかできない、ではない。**作成年を記録** |
| E | **鉄板3本あれば一流**（上岡龍太郎）。数より質。サンド「ピザのデリバリー」=披露100回超の一生モノ（調査3 §2） | 少数精鋭が正義。**枠は絞る**（多数薄くより数本厚く） |
| F | **大会は場でネタを使い分ける**。予選=手堅い勝負ネタ/決勝=2本制（1本目=空気掌握/2本目=温存の最強 or 意表）。**"また同じ"回避**・**勝負ネタの出しどころ**戦略（調査4 §3/§4・調査3 §4） | **大会でどのネタを出すか選ぶ**意思決定。同系統で同じネタ連投は目減り |
| G | **型は2軸が混在**。演じ方軸（しゃべくり/コント＝1本1型が自然）＋技法軸（伏線回収・天丼・つかみ＝本来"部品"で1本に併存しうる）。「そのネタが最も勝負している要素」と定義すれば1本1型は自然（調査5 §3/§4） | **型はネタが持つ1属性**（年単位の選択ではない）。「最も勝負している要素」1つを持たせる |

> **核**: 「作る → ライブで磨く（客反応でウケが可視化・完成度↑）→ 持ちネタに貯まる（年跨ぎ持続・古いネタも使える）→ 大会で反応の良いネタを選んで出す」の**持続ループ**。型は年の選択でなく個体の属性。

## 2. ネタのデータ構造【仮】

### 2-1. `Neta` 個体（GameCore・新規 `Neta.swift`）

`exp*`（`GameState.swift:30-38`）や `ChoiceEventEffect.swift` と同じ**純Swift・RandomSource非依存・Codable**。数値は全て【仮】。

```swift
// Neta.swift（新規・GameCore）— RandomSource を一切呼ばない純データ＋純関数。
// golden非干渉: このファイルのどの関数も乱数を引かない・GameEngine.perform の式に触れない（Phase 0）。
public enum NetaKata: String, Codable, CaseIterable {   // 型＝ネタが持つ1属性（年の選択ではない・§1-G）
    case 王道しゃべくり   // 会話の純度・間
    case 関係性         // 二人の歴史・掛け合い
    case 伏線回収        // 多層構成
    case リターン        // 天丼・帰る場所（反復）
    case 非定型         // 新しい型の発明（実験）
    case 瞬発          // 畳みかけ・勢い押し
    case 華先行         // つかみ・存在感で持つ
    // 注: judge_preference_research_v0.md §5／multiyear §2-3 の7型と同一。旧v0の5型を7型に更新（§8）。
}

public struct Neta: Codable, Identifiable {
    public let id: Int                 // 生成連番（セーブ安定・表示ソート用）
    public var name: String            // 題材名（自動生成 or プレイヤー命名・一般名詞のみ／§7-決点3）
    public let kata: NetaKata          // 型＝1属性（作成時に確定・§1-G「最も勝負している要素」）
    public var polish: Double          // 完成度 0..100（稽古・ライブで上がる）
    public var buzz: Double            // ウケ/客反応 0..100（ライブでの手応えの移動平均・§3-2）
    public let bornYear: Int           // 作成年（芸歴年・§1-D「古いネタも使える」の証拠）
    public var stageCount: Int         // 通算ライブ数（"漫才はドラクエ"の経験値・§1-B）
    // 戦績（§5-3 ネタ帳の"育った実感"用・非スコア）
    public var record: [NetaStamp]     // 「3回戦・通過」等のスタンプ列
    public var lastUsedTaikaiYear: Int?  // 同系統での連投目減り判定に使う年（§4-2）

    public init(id: Int, name: String, kata: NetaKata, bornYear: Int) {
        self.id = id; self.name = name; self.kata = kata
        self.polish = 30; self.buzz = 0; self.bornYear = bornYear
        self.stageCount = 0; self.record = []; self.lastUsedTaikaiYear = nil
    }
}

public struct NetaStamp: Codable {     // 戦績スタンプ（表示専用）
    public let year: Int
    public let stage: String           // "GP3回戦" 等（StageResult.name をそのまま）
    public let passed: Bool
}
```

- **完成度 `polish` と ウケ `buzz` を分ける理由**（調査2 §3・調査C）: `polish`=磨き込みの累積（下がらない資産）。`buzz`=**直近のライブでどれだけウケたか**（客層・出来で振れる直近値）。実在の「同じネタでも会場で反応が変わる」＝ `buzz` は振れるが `polish` は削り込んだぶん残る。UIで別に見せると「磨けた（polish）／今ウケてる（buzz）」の2軸で手応えが立つ。

### 2-2. 持ちネタ帳（GameState 拡張・golden安全）

`GameState`（`GameState.swift:4-50`）に**フィールドを足すだけ**。`exp*` 追加（同 30-38）と完全に同型＝Codable自動・`CodableTests.swift:11` の往復に1行足すだけで担保。**perform（`GameEngine.swift:145`）はこの配列を読まない＝golden不変**。

```swift
// GameState.swift への追加（Phase 0）— exp* と同じ「表示・資産の器」。perform は参照しない。
public var netas: [Neta] = []          // 持ちネタ（レパートリー・年跨ぎ持続）
public var nextNetaID: Int = 0         // 生成連番の採番（決定論・乱数非依存）
public var selectedNetaID: Int? = nil  // 大会に「今かける」ネタ（自由週にデフォルト=前回踏襲・§4）
```

- **枠は最大4本、5本目作成時に1本を封印**【仮】（旧v0 §2 踏襲・調査E「鉄板3本＝数より質」）。封印は削除でなく `archivedNetas: [Neta]` へ移す（§5-3「お蔵入りページ」）。
- **なぜ配列で golden 安全か**: `netas` は `perform` の入力（`jitsuryoku`+`compat`+roll・`GameEngine.swift:158`）に一切入らない。追加は「セーブに乗る器」を増やすだけ。乱数消費（perform の2draw＝`GameEngine.swift:149-150`／rollOffer＝`139-140`／injury＝`WeekRunner.swift:228`）の**順序も回数も変わらない**。

## 3. アクション接続 —「作る」「磨く」を既存5稽古に接地

既存の稽古5種（`GameConfig.swift:150-155`／UIカタログ `CommandData.swift:53-59`）の**能力効果は据え置き**、そこに**ネタ個体への作用を"重ねる"**。旧v0 §3 の方針を、7型・buzz対応に更新。

### 3-1. 「作る」＝`ネタ作り`（`GameConfig.swift:151`・現 発想+3/センス+1）

| 現行（据え置き） | ネタ個体への作用（新・Phase 0） |
|---|---|
| 発想+3・センス+1（`creditTraining` 経由で粒） | **選択中ネタが無ければ新ネタ生成**（`polish=30`・型を選択・`nextNetaID` を採番）。**あれば改稿＝`polish += 8`**【仮】 |

- **型の選択タイミング**（§1-G「1属性」の担保）: **新ネタ生成時のみ**、型を1つ選ぶ（「このネタで最も勝負する要素は?」）。**年の選択ではない・ネタごと**。旧Fable案（年始に1つ）との決定的な違いはここ。
- **UI**: `ネタ作り` カードをタップ→（持ちネタに空きがあり選択中ネタが無い/新規を選んだ場合）**型選択の小シート**を1枚挟む→生成。改稿の場合は選択中ネタに +polish。**3秒動線を壊さないため、デフォルトは「選択中ネタを改稿」**（型シートは新規時のみ）。

### 3-2. 「磨く」＝`ネタ見せ会`／`フリーライブ`（客前でかける＝ウケで上がる）

実在の核（調査2 §2/§6「劇場でかけるほど完成度が上がる／客反応が羅針盤」）を、**この2アクションに背負わせる**。

| アクション | 現行（据え置き） | ネタ個体への作用（新・Phase 0） |
|---|---|---|
| `ネタ見せ会`（`GameConfig.swift:152`・表現+6/メンタル+3・有料80,000） | 舞台稽古＝表現・メンタル | 選択中ネタに **`polish += 12`・`stageCount += 1`**。**この回の"ウケ"を算出（下記）→ `buzz` を移動平均で更新**【仮】 |
| `フリーライブ`（`GameConfig.swift:155`・華+3/表現+1・無料・知名度+1） | 客前で場数 | 選択中ネタに **`polish += 6`・`stageCount += 1`**・**ウケ算出→`buzz` 更新**。**営業=数を踏む場**（安いぶん磨きは緩め） |

- **ウケ（客反応）の算出は決定論・純関数＝golden非干渉**（Phase 0の生命線）:

```swift
// Neta.swift — ライブ1回のウケ。RandomSource を呼ばない純関数（golden非干渉）。
// 実在: 出来（今の実力）× ネタの完成度で、その日の客の沸きが決まる（調査2 §3）。会場差＝将来枠。
public static func liveBuzz(state s: GameState, neta: Neta, config: GameConfig) -> Double {
    // 現在の演技実力（perform と同じ重み・GameEngine.jitsuryoku を"読むだけ"＝式は変えない）
    let power = GameEngine.jitsuryoku(s, config: config)   // 参照のみ・perform の乱数には触れない
    // 完成度が高いほど客に伝わる。0..100 に正規化（係数は【仮】・GameConfig集約）
    let raw = power * config.netaBuzzPowerW + neta.polish * config.netaBuzzPolishW
    return GameEngine.clamp(raw, 0, 100)
}
// buzz の更新は移動平均（直近のライブほど効く・振れは残す）:
//   neta.buzz = neta.buzz * (1 - α) + liveBuzz(...) * α   （α=0.4【仮】）
```

  - **なぜ乱数を引かないか**: ここで `rng.nextUniform()` を引くと**消費順が動き 3年 golden が壊れる**（`WeekRunner.swift:228` の injury 抽選と同じ位置＝RNGは消費順が正典）。Phase 0 は**出来ブレを入れず、実力×完成度の決定論値**にする。「その日ごとの客のムラ」を乱数で入れたくなったら→**§6-2 で止めて「4.8相談（規律A）」**。
  - **実装位置**: `applyTraining`（`GameEngine.swift:80`）は選択中ネタIDを知らないため、**ネタ作用は WeekRunner 側で "行動確定の直後・RNG非消費の区画" に置く**——`applyAllocation`（`WeekRunner.swift:338`）/`applyEventEffects`（同 330）と**同じ"権威stateへの純適用"経路**を新設（`applyNetaWork(training:)`）。この区画は**乱数を一切呼ばない**ことがコメントで保証されている領域（同 329-336）＝そこに相乗りする。

### 3-3. `ネタ合わせ`／`ランニング・サウナ` は据え置き（ネタ非依存）

- `ネタ合わせ`（`GameConfig.swift:153`・センス+3/相性+1）＝「ネタより二人の呼吸」の受け皿。ネタ個体には触らない（旧v0 §3 の思想を維持）。
- `ランニング・サウナ`（同 154・メンタル+6）＝体調・メンタル。ネタ非依存。

> **接続の要旨**: `ネタ作り`＝生む/改稿（polish）。`ネタ見せ会`/`フリーライブ`＝**客前でかける＝ウケ算出＋polish**（実在の"ライブ磨き"）。`ネタ合わせ`/`ランニング`＝ネタ非依存の土台。**全て既存アクションに乗り、能力効果は据え置き。ネタ作用は RNG非消費の純適用区画に置く＝golden不変**。

## 4. 大会でのネタ選択 —「反応の良いネタを選んで出す」

### 4-1. 選ぶ場所（既存の大会/GP入力にネタ選択を"隣接"させる）

現在、大会週は `tournamentDecision`（`WeekRunner.swift:44`・出る/移動手段）、GP回戦・決勝は `resolveAuto`（同 156・入力不要の自動消化）。ここに**「どのネタで出るか」を差し込む**。

- **選択の実体は `state.selectedNetaID`**（§2-2）。**大会週に入る前の自由週にプレイヤーが選んでおく**（デフォルト＝前回踏襲＝旧v0 §8-1「3秒ルール」）。大会入力画面には**"今かけるネタ"の確認ピル**だけ出す（タップで持ちネタ帳を開いて差し替え可）。
- **これは判定に影響しない選択（Phase 0）**＝ `perform` に渡らない。**"どれで出たか"は戦績スタンプ（§5-3）と講評/見せ札の演出材料（§4-3）にのみ効く**。「反応の良いネタを選ぶ」体験は Phase 0 で**演出・戦績として先に立ち上げ**、スコア寄与は Phase 1。

### 4-2. 決勝の2本制・同系統の連投目減り（実在Fの再現）

- **決勝2本制**（調査F・調査4 §3/§4）: 決勝（`WeekRunner.swift:192` `.final`）に入る時、**2本目のネタも要求**（`selectedNetaID2`）。**1本目=空気掌握／2本目=温存の最強 or 意表**という実在の役割を、**演出（見せ札の"1本目/2本目"ラベル・§4-3）と戦績**で見せる。Phase 0では**スコア無関係**（旧v0 §4「2本目60未満で-3」は Phase 1 の話＝§6-1へ隔離）。
- **同系統で同じネタ連投は"目減り"**（調査F「また同じ回避」・調査3 §4「勝負ネタの消耗」）: `lastUsedTaikaiYear` と大会系統で判定。**Phase 0では「講評が"また同じネタか"と触れる」演出だけ**（`buzz` 表示を少し渋くしてもよいが**スコアには乗せない**）。実際に完成度補正を半減する（旧v0 §4）のは Phase 1。

### 4-3. 大会でのネタ選択UI【仮】

- **予選/大会入口**（`tournamentDecision` 画面）: 出る/見送り・移動手段の上に、**"今夜かけるネタ"カード**（型アイコン＋題材名＋`polish`ミニバー＋`buzz`ドット）。タップ→持ちネタ帳オーバーレイで差し替え。
- **決勝入口**（`gpFinal` 演出の前・`FinalsPresentationView` 冒頭 `lotBeat`＝`FinalsPresentationView.swift:70` の直前）: **2枠**（1本目/2本目）。「温存していたのはどっちだ」の一枚をここで置く。
- **見せ札への波及**（非スコア・演出）: `FinalsPresentationView` の `judgeCard`（同 106）に、**選んだネタの型を"審査員の◎/×"で色づけるラベルを1語足せる**——§4-4 の相性表を引いて「この審査員に刺さる型/取りこぼす型」を演出。**現状のtiltは能力値ベース（同 232-244）で正しい**。型ラベルは**neta_system 実装後に足す**（multiyear §2-5 の指示どおり・**tiltの数式は変えない**）。

### 4-4. 7審査員 × 型の相性（固定嗜好・非スコア・演出材料）

**"年単位の選択"ではなく"各ネタが持つ型の属性"がこの表に当たる**（§1-G）。表は `multiyear_and_judge_preference_integration_v0.md` §2-3／`judge_preference_research_v0.md` §5 の正本をそのまま採用（◎強く刺さる ○刺さる △鈍い ×取りこぼされやすい・全て【仮】）:

| 型 ＼ 審査員 | 天堂寺(王道) | 花園(情) | 卯月(構成) | 目白(設計) | 白波(熱) | 神楽坂(話芸) | 音羽(華) |

## 9. 要オーナー決点

> **決点1（型の数・7 vs 5）**: 本書は**7型**（王道しゃべくり/関係性/伏線回収/リターン/非定型/瞬発/華先行）を採る＝審査員相性表（§4-4）と1対1。旧v0の5型より1枚のネタが背負う個性が増えるが、UIの型アイコンが7種になる。**7でよいか／5に減らすか**。推奨: 7（相性表とドラマ密度のため）。

> **決点2（枠数・4本 vs 3本）**: 実在は「鉄板3本あれば一流」（調査E）。**持ちネタ枠を4本にするか3本にするか**。3本は封印判断が早く来て緊張が増す／4本は"試作をもう1本"の余白がある。推奨: 4（旧v0踏襲・封印の緊張は5本目作成時に立つ）。

> **決点3（ネタ命名・自動 vs プレイヤー）**: 題材名を**自動生成**（一般名詞リスト:「コンビニ」「家電量販店」「結婚式の余興」等・実在ネタと偶然一致しても問題ない語に限定）にするか、**プレイヤー命名**を許すか。推奨: 自動をデフォルト＋任意で改名可（`Neta.name` は var）。命名リストは `name_generator` と別管理。

> **決点4（Phase 1 に進むか＝ネタを勝敗に効かせるか）**: **§6-2 の(a)(b)＝完成度・ウケのスコア補正と2本制ペナルティを入れるか**。入れると「反応の良いネタを選ぶ」が本当に勝率に効く（実在Fの戦略が生きる）。代償は**規律A（golden全再生成）＋sim較正**（`exp_neta.py` は実測済み＝係数0.04で中央値±0年・旧v0 §6-B）。推奨: **Phase 0 を先に出し、体感を見てから Phase 1 を判断**。**新しい乱数を引く方向（会場差でウケが振れる等）に踏み込むなら、その時点で止めて4.8相談（規律A）**。

> **決点5（型×審査員相性をスコアに乗せるか＝§6-2(c)）**: `judge_design §0`「審査は総合点・違いは表示のみ」の**格上げ**に当たる（multiyear §2-6 と同一の未決）。推奨: **当面乗せない**（表示・講評で"読み"の8割は成立し、golden全再測＋endgame/dynasty 再アンカーの代償が重い）。**オーナー判断待ち**。
