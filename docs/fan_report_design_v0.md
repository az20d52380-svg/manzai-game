# ファンレポ機能設計 v0 — 予選は点数が出ない・レポを見てから合否・「研究される」対策

- 日付: 2026-07-05
- 位置づけ: 実在の賞レース予選の観戦レポ文化（ブログ・SNS）をリサーチし、(1)予選の結果演出を「本番→ファンレポ→合否」の順に再設計 (2)レポ文の型分析と量産 (3)「一度決勝に行くと研究されて上がりにくい」現象の機構化——をまとめる。固有名は架空、数値は全て【仮】。

## 0. リサーチ — 実在の予選レポ文化の傾向

- **予選（1回戦〜準決勝）は点数が公表されない。合否のみ**。だからファンのレポが「観測手段」として文化になっている（本作もこれに従う: **予選で数値スコアをUIに出さない**）
- レポの定番フォーマット: **ウケ量を10段階などで主観評価**＋会場の空気＋一言感想。「ネタバレなし」が礼儀（ネタの中身は書かず、ウケと手応えだけ書く）
- 頻出の言い回し: 「会場が揺れた」「ドカンと来てた」「ハネてた」「刺さってなかった」「空気が重かった」「出順が不利」「手応えあり」「今年は仕上がってる」
- **常連・決勝経験組への特有の言説**: 「面白いけど新鮮味が…」「もう見慣れてしまった」「対策されてる感」「去年の衝撃はない」——**システムがバレる/飽きられる現象はファン言説として実在する**
- レポは主観なので**外れる**。「あんなにウケてたのに落ちた」「静かだったのに通った」も定番の話題（審査員とファンの目線差）

## 1. 機能設計 — 結果発表の演出順序を変える

### 予選（GP1回戦〜準決勝・道中大会）

1. 本番（笑い波形メーター）
2. **ファンレポ 2〜4本**（SNS風カードで流れる。タップ送り）
3. 溜め → 合否スタンプ

- **点数は出さない**（実在準拠）。プレイヤーはレポの空気から合否を予感してスタンプを見る——「レポは良かったのに…」「渋いレポだったが通った！」の感情の揺れが1回戦ぶんの物語になる
- 決勝のみ審査員の表示点＋講評（judge_design）。**「予選=群衆の声／決勝=審査員の声」**の対比が世界観になる

### レポの生成規則【仮】

- 生成キー: `(内部スコア帯, ハマった夜の有無, 知名度帯, 常連度, 直近の型)` からプールを引く
- **確定的なネタバレをしない**: スコア帯とレポの明るさは相関させるが、**20%はミスリード枠**（高スコアでも渋いレポ／低スコアでも熱いレポを混ぜる）。レポ=観客の主観、合否=審査、のズレを仕様として残す
- ハマった夜は必ずレポに痕跡を残す（「今日の〈コンビ名〉は何かが起きてた」）——**運を"見える運"にする**装置（canonical_v2_spec §1の勝負の運のUI課題への回答）
- 同一文の再表示はキャリア内禁止（審査員コメントと同じ直近使用ID方式）

## 3. 「一度決勝に行くと上がりにくい」の機構化【研究されるデバフ】

リサーチ根拠: 常連・決勝経験組への「新鮮味がない」「対策されてる」「見慣れた」というファン言説は実在の定番（§0）。これを挑戦者側の飽きられ（dynasty §3・実測済み）の**決勝版**として機構化する:

- **研究されるデバフ【仮】**: 決勝進出の翌年から、**同じネタの型**（neta_system の型5種）で挑む準決勝・決勝の実効ライン+3
- **解除条件**: (a)**型を変える**（ネタ資産システム連動・本命。レポ#33「型を変えてきた！」が付く） (b)2年その舞台に出ない (c)優勝する（王者は別の物語=王者編の飽きられへ接続）
- 設計意図: 「決勝の常連になって足踏み」ではなく**「決勝に行ったからこそ、変わらざるを得ない」**という成長の強制イベントにする。実在の決勝経験組が型を更新して再浮上する物語と同型
- バランス根拠: dynasty §3-Bで「デバフ-3は解除手段とセットなら分布非破壊（基準と完全一致）」を実測済み。同じ構造なので安全側。**ネタ型（フェーズ1）実装時に exp_challenger の枠組みで再計測**すること
- MVP（1年版）は対象外（決勝進出→翌年が存在しない）

## 4. 実装メモ

- レポ本文はUI層リソース。GameCoreは生成キー `(帯, ハマ, 知名度帯, 常連度, 型)` を返すだけ（ルール1の分離）
- 「常連度」= 過去の決勝進出回数と同一回戦の連続敗退数から算出【仮】
