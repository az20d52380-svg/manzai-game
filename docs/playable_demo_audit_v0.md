## 3. 面白さ・感情の核はどこにあり、どこで削がれているか

### 3-1. 核がある場所（守るべき4点）

1. **判の1.6秒**（`TournamentResultView`：波形→静止→押印→半紙→星の段階開示）。この15秒がビルドで一番ゲームらしい。負けにも同じ物量、の設計がちゃんと実装されている。
2. **照準の逆算ループ**：最下帯「大会までN週」＋`AllocationView`「つぎの本番」カード（地力バー・仮置きゴースト・朱の要求ライン）。「あと2週、どこに注ぐか」が今作の意思決定の芯であり、経験点導入でようやく**噛む歯車が2枚**（どの稽古で稼ぐか×どこへ注ぐか）になった。
3. **粒→注ぐ→伸びる**：+N粒チップ→バッジ→リビール。オーナー自身が実機で報告した「伸びわずか」の味気なさへの正面回答で、方向は正しい（目視待ち）。
4. **器満了の年内転調**：満了後、稽古カードが「満」判に変わり、年後半が自然に「金・体力・相性・本番」の別ゲームに切り替わる。意図的設計＋較正済みで、1年に起承転結を作る良い骨。

### 3-2. 削がれている場所（5点・深刻度順）

1. **年の締めに物語が無い**。§1-3のとおり全ランが負けで終わるのに、`YearResultView`は判と統計だけ。しかも**谷口の年末独白は以前のビルドには在った**（`f763e97`で配線→S6全面刷新`e347629`で削除＝退行）。corpus側には`yearEnd.躍進/停滞/貧乏`が完成済み、分岐拡張の設計もオーナー決定済み（HANDOFF §A-5）。**デモの最後の画面が、書き上がった言葉を一番待っている。**
2. **中腹の無言**。週16〜26は本番ゼロの11週（オファー抽選のみ）。その間プレイヤーが読む言葉は平常3本＋「そろそろ○○か。×、あと少し上げておきたい」の単一テンプレの反復。`fun_flow_review`が自認した中だるみが、1年版では言葉の在庫切れとして最初に露出する。**在庫は既に書いてある**（§8量産済み・採点済み）。配線だけが無い。
3. **悔しさの翌週の温度事故**。justLost未配線のため、大会敗退の翌週に「調子は悪くない。このまま積み上げよう。」が出うる。NIGHT_LOG（0003）が予言したとおりの事故がコード上いまも可能。負け→翌週の一言こそ「下積み→悔しさ」の芯。
4. **勝負の運が不可視**。`perform`はscoreを計算するが`StageResult`に載せず捨てている＝ハマった夜（+12）が出ても**プレイヤーには一生分からない**。canonical §1は「ハマは見える運として必ず可視化する」と約束している。波形は通過/敗退の2種しか形が無く、星も能力導出（その夜の出来と無関係）——「その夜」が存在しない。デモ必須ではないが、感情の核（決勝は特別な場所＝運と実力の劇）の可視化がゼロという自覚は要る。
5. 記録の手触り：きろくページの判ミニチュアに体調ダウンが「敗退」で混ざる（§1-4-5）。

### 3-3. 過剰・散らかりの判定（忖度なし）

- **経験点システムは過剰実装ではない**。2度却下→オーナー自身の実機体感→反証（simスパイクNO-GO/GO実測）→採用、という却下履歴への正面手続きを踏んでおり、機構もρ=0に縮退させた最小形。本筋（成長の実感）への直撃であり、寄り道ではない。
- 散らかりとして残っているのは**休眠物と嘘**：無機能の通知許諾／ρ=0で意味を失った二区画UIの残骸（共通0チップ・説明文）／`ChoiceEventEffect`＋`applyEventEffects`（GameCoreの器とテストのみ・呼び出しゼロ＝無害だが誰も使っていない）／`reaction`死に辞書／才能表示の嘘。どれも「消す/隠す/正直にする」で足り、作り直しは不要。
- 強いて言えば`S6bView`（勇退ED・155行）は現状DEBUG専用到達だが、優勝の価値を先に作っておく投資として許容範囲。これ以上「届かない側」を膨らませないこと。

## 4. 経験点機能の直後に作るべきもの（1つ）

### 推奨：**声の最小配線パック**（1日級【仮】・golden安全・新規機構ゼロ）

理由（3行）：
(i) デモの合否は検証①「48手が単調にならないか」で決まり、その単調さの最初の露出点が**最頻画面の言葉の在庫切れ**（§3-2-2）——ここに効く最安の一手。
(ii) 在庫500行超は執筆・採点・検品まで済んでおり、残作業は配列差し替え中心＝**RNG非消費でgolden不変**、swift testは緑のまま。工数対体験の比が候補中で桁違い。
(iii) これは新しい優先順位ではなく、`game_reality_check` §6が7/5に定めた「本より先にやること1番」の実行。プロジェクト自身の結論に返すだけ。

### 反証条件（これが出たら推奨を捨てる/入れ替える）

1. **次のビルドを監督なし・複数セッションで他人に渡す予定が先にあるなら、0039セーブを先に**（P1⇄P2入替）。監督つき10分テストならこの順で良い。
