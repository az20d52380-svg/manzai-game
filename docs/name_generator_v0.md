# コンビ名ジェネレーター v0 — 固有名詞の組み合わせで毎周ランダム・被らない架空名

> **【正典移行バナー・2026-07-05オーナー確定】**
> `content_batch5_v0.md` §2／§2追補の**固定100組リスト**は、決勝敗北時の優勝者・TOP3補充・回戦すれ違い演出で使う「コンビ名プール」だった。
> オーナー新方針「**コンビ名は語の組み合わせで毎周ランダム生成**（固有名詞＋固有名詞 もしくは 固有名詞のみ）」により、**本書のジェネレーターが正典（名前の一次供給源）**になる。固定100組は**(a) 語バンクの種語**と**(b) 生成失敗時のフォールバック**として存続（履歴・非破棄）。
> `champion_selection_v1.md` §3/§5・`relationship_champions_v0.md` §5-3 の「name pool から引く」記述は、本書の `genDuoName()` 呼び出しに置き換わる。

- 日付: 2026-07-05
- 位置づけ: 周回ごとに、役柄テンプレ（同期／後輩／先輩）とモブ（知らない組）へ**架空コンビ名を決定的に生成**する。暗記不可（`runSeed` で毎周シャッフル）・衝突ほぼゼロ（組み合わせ爆発）・セーブ/めくり再現性あり（純関数）。
- **バランス数値・golden に一切触れない**（§5 で機構的に保証。`champHash` 独立・gameplay RNG を1消費もしない）。
- 固有名は全て架空。実在コンビ・人物・番組・SNSを想起させる語は禁止。「笑」を SNS スラング用法（(笑)・w）で使わない（命名基準 `judge_design_v0.md` §10-D）。

## 0. なぜジェネレーターか（オーナーの着眼）

固定リスト（100組）だと、周回を重ねると**同じ名前が繰り返し出て暗記される**。オーナー案「**固有名詞＋固有名詞／固有名詞のみ**」を採ると、語の組み合わせで名前空間が一気に広がり、**被らない**：

- 語バンク A（約40語）× 語バンク B（約40語）× 接続3型 ＝ **約4,800通り**（2語型だけで）
- ＋ 単語型（固有名詞のみ・約120語）＋ 接尾型 ＝ **合計 概算6,000通り超**
- 1周で使う名前は最大でも数十。**同一周回内の衝突は非復元抽選で完全排除**、**周回間の再会（同じ名前が別周で出る）確率も実用上ゼロ**。

プレイヤーが覚えるのは「関係の色」と「物語のリズム」だけになり、**名前と結果の対応は毎周崩れる**（`champion_selection_v1 §4-3` の暗記不可担保と合致）。

## 1. 生成パターン（4型・オーナー案の実装）

`〈A〉`＝語バンクから引く語。パターンは `champHash` で選ぶ。

| 型 | 形 | 例（種語から） | 重み【仮】 |
|---|---|---|---|
| P1 2語連結 | `〈A〉〈B〉` | 深夜＋グラタン → 深夜グラタン | 30 |
| P2 の接続 | `〈A〉の〈B〉` | 岬＋灯台 → 岬の灯台 | 25 |
| P3 単語のみ | `〈S〉` | 温故知新／みそしるず | 25 |
| P4 語＋接尾 | `〈A〉〈接尾〉` | 乾電池＋兄弟 → 乾電池兄弟 | 20 |

- 重みは【仮】（体感調整はまず頻度・演出で・規律B-5）。合計100で正規化。
- P1/P2/P4 の `〈A〉`〈B〉は**別カテゴリから引く**のを基本にして語感を散らす（同カテゴリ連結は単調。§2-2 の相性ルール）。

## 3. 決定的生成（`champHash` 独立・機種非依存）

`champion_selection_v1 §4-1` と同一の `champHash`（splitmix64・整数のみ・`&*`/`&+` オーバーフロー）を共有。**用途 salt を分離**して衝突回避。

```swift
extension Salt {
    static let nameGenPattern: UInt64 = 0x5A17_0000_0000_0020  // パターン選択
    static let nameGenWordA:   UInt64 = 0x5A17_0000_0000_0021  // 語A
    static let nameGenWordB:   UInt64 = 0x5A17_0000_0000_0022  // 語B / 接尾 / 単語
}

struct NameGenConfig {                 // 演出層・golden対象外（§5）
    var patternWeights: [Int] = [30, 25, 25, 20]   // P1,P2,P3,P4【仮】
    var maxProbe: Int = 24                          // 衝突/NG時の再ハッシュ上限
}

// slot = その周回で通し番号（役柄名は §3 の割当順、モブは出現順）。
// probe = 衝突/NG 時のリハッシュカウンタ（決定的に増やす）。
func genDuoName(runSeed: UInt64, slot: Int, used: Set<String>,
                cfg: NameGenConfig) -> String {
    for probe in 0..<cfg.maxProbe {
        // 年は 0 固定（名前は周回定数）。slot と probe を salt に畳み込む。
        let base = UInt64(bitPattern: Int64(slot &* 131 &+ probe))
        let hp = champHash(runSeed, 0, Salt.nameGenPattern &+ base)
        let pat = weightedPick(cfg.patternWeights, hp)          // 0..3

        let ha = champHash(runSeed, 0, Salt.nameGenWordA &+ base)
        let hb = champHash(runSeed, 0, Salt.nameGenWordB &+ base)

        let name: String
        switch pat {
        case 0: name = pickA(ha) + pickB(hb, avoidCat: catOf(ha))      // P1
        case 1: name = pickA(ha) + "の" + pickB(hb, avoidCat: catOf(ha)) // P2
        case 2: name = pickSingle(ha)                                  // P3: WORD_SINGLE ∪ WORD_CONCEPT から1語
        default: name = pickA(ha) + SUFFIX[Int(hb % UInt64(SUFFIX.count))] // P4
        }
        if !used.contains(name) && passesNG(name) { return name }      // §4/§6
        // 衝突 or NG → probe++ で決定的に別候補へ（機種非依存・再現可能）
    }
    // フォールバックも used を尊重（不変条件を破らない）:
    // 固定100組を hash 起点から巡回し、未使用かつNG通過の最初の1件を返す。
    // キャスト数 ≪ 100 ゆえ必ず空きがある（precondition で落とさない・§5-1）。
    let start = Int(champHash(runSeed, 0, UInt64(slot)) % UInt64(FALLBACK_POOL.count))
    for k in 0..<FALLBACK_POOL.count {
        let cand = FALLBACK_POOL[(start + k) % FALLBACK_POOL.count]
        if !used.contains(cand) && passesNG(cand) { return cand }
    }
    return FALLBACK_POOL[start]   // 論理的に到達不能（used < 100 を保証・§4）
}
```

- ヘルパの対象バンク（§7で確定）: `pickA`/`pickB`＝FOOD/SCENE/TOOL/ANIMAL/CONCEPT から（`catOf` で引いたカテゴリを記録し §2-2 の別カテゴリ制約 `avoidCat` に使う）／`pickSingle`＝`WORD_SINGLE ∪ WORD_CONCEPT`／`weightedPick`＝patternWeights。**すべて整数演算のみ**（浮動小数を使わない＝機種非依存）。
- **同一 `runSeed`＋同一 `slot`＋同一 `used` → 必ず同じ名前**（純関数・save/めくり再現）。
- `used` は「その周回でここまでに確定した全名前」。役柄名割当（`champion_selection_v1 §3`）→モブ生成（決勝TOP3補充・レポ）の順で **1つの `used` 集合を引き継ぐ**（消費順は正典・§5-2）。

## 4. 衝突排除とフォールバック

1. **同一周回内**: `used` 集合で非復元。生成名が既出なら `probe++` で決定的に別候補へ。→ 同じ画面・同じ周回で同名が出ない。
2. **`maxProbe` 到達**（語バンクが痩せている等の保険）: `FALLBACK_POOL`＝`content_batch5` の100組（固定・凍結index）から `runSeed×slot` で引く。実運用では到達しない想定（名前空間6,000超）だが、**決定的な最終手段**を必ず持つ（precondition で落とさない）。
3. **周回間**: 別 `runSeed` は別系列。数千規模の空間ゆえ「前周と同じ名前」はまれ。仮に一致しても**暗記の手掛かりにならない**（結果との対応が毎周崩れる・`champion_selection §4-3`）。

## 5. 非バランス・周回安定（`champion_selection §6` と同じ保証）

### 5-1. golden 不変
- 名前生成は **gameplay の注入RNGを1消費もしない**。`champHash`（`runSeed`＋salt）だけで閉じる。→ 週送り/稽古/大会の**乱数消費順に触れない＝3年 golden ビット一致は不変**（規律A）。`tools/*.py` 正典・`swift test` に影響なし。
- `NameGenConfig`（重み・maxProbe）は**演出層の定数**。`sim_career.py`／バランス golden の対象外。全て【仮】。

### 5-2. 消費順＝正典
- salt は `Salt.nameGenPattern/WordA/WordB`＋`slot`＋`probe` で一意。**slot の採番順（役柄名割当→モブ）を変えると同一 `runSeed` でも別名になる**。順序は数式と同格の正典（規律A-2）。

### 5-3. save/めくり整合
- 永続化は **`runSeed` のみ**。名前は `runSeed` から再導出可能（純関数）。決勝画面でセーブ→ロードしても同じ名前・同じ優勝者。

## 6. NG フィルタ（`passesNG`・リリース前弁護士確認と二段構え）

生成名・種語の両方に適用（種は登録時、生成名は出力時）。

1. **実在想起の排除**: 実在コンビ・芸人・番組・企業・商品・SNSサービスの名称／その音・字面に近い組み合わせを NG リストで弾く（例リストは `judge_design §10-D`・`roadmap §4` の弁護士確認リストに集約）。
2. **「笑」スラング禁止**: 「笑」を SNS スラング用法（(笑)・草・w）で含む語・並びを禁止。
3. **既存NPC固定名の予約**: 旧固定名（雨宿り 等・`content_batch5 §2` の除外2組相当）を生成語に混ぜない（NPC と役柄名の混同防止）。
4. **不適切語**: 侮蔑・差別・下品な連結を弾く（種バンク登録時に人手チェック＋語ブラックリスト）。
5. **語感の破綻**: 読めない／意味が通らない連結（同語反復「グラタングラタン」等）を軽く弾く（同語 A==B 禁止・§2-2 のカテゴリ相性で緩和）。

> **【設計上の懸念】** 組み合わせ生成は「たまたま実在名や不適切語に化ける」リスクが固定リストより高い。**リリース前に、代表 `runSeed` を数百流して生成名を全ダンプ→弁護士確認リストへ**（`roadmap §4`）。NG リストは生成側の第一防波堤で、最終責任は人手レビュー。
