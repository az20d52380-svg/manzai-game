## 2-B. 成長上限の可視化と「使い切った後」の出口【2026-07-05・red_team §1-3対応】

問題: 成長予算（正典v2・キャリア累計）を使い切ると演技系4能力（センス/発想/表現/華）の稽古が伸びない。§2の「効果を事前表示しない」ままだと、伸びない稽古が「壊れてる？」に見える。

方針: **上限は隠さない。ただし"効果量の事前表示"はしない**（§2の考える楽しみは維持）。隠すのは各行動の数値、見せるのは「今、伸びしろが残っているか」の状態1点——両立する。

### コード対応（GameEngine.add / GameState・ロジック変更ゼロで読める）

- 判定量: `残り伸びしろ = (growthBudget ?? ∞) − growthUsed`（演技4能力ぶんの残予算。**メンタル・相性は予算外なので含めない**——add: `a != .メンタル`／相性は別系統の StatKey）
- 「使い切り」トリガ: `残り伸びしろ` が標準稽古1回の実効上昇（重み換算で概ね1.0〜1.5）を下回ったら点灯【仮閾値・sim後に確定】

### 表示（2箇所・数値は出さない）

1. **S5ネタ帳**: 能力欄の上に「伸びしろ ▓▓▓░░」ゲージ（残予算を当年上限カーブ値で正規化・繰越で満杯超過分はキャップ）。ゼロ帯は灰＋「今季ぶんは出し切った」。稽古のたびゲージがコース目盛りで減るのを差分ポップ（§6）で見せる＝「伸びは有限」を体で理解
2. **S2週メイン**: ゲージがゼロ帯に入った**最初の週だけ**、イベント帯に谷口の台詞（毎週は出さない・くどさ回避）

### 谷口の台詞【仮・dialogue_design register】

- 点灯時: 「**今年ぶんの伸びは、もう出し切ったな。……ここから先は、数字やない所で差をつけるんよ**」
  - ※残予算はキャリア累計のため前年繰越があり得る。繰越が残る年は別文面「まだ伸びしろは残っとる。焦らんでええ」。"今年ぶん"表現は繰越ゼロの通常時のみ
- 出口提示: 「メンタル鍛えたら本番ブレんくなる。相性上げたら素点が乗る。ネタの鮮度もある。──伸びが止まっても、やることは残っとる」

### 「使い切った後」の別の出口（既存コードで既に有効）

| 出口 | 仕組み（既存コード） | 効果 |
|---|---|---|
| **メンタル** | 成長予算の対象外。ランニング・サウナで+6 | ブレ幅 B=5+15×(1−m/100) を狭める＝本番が安定（平均で越えるラインを確実に） |
| **相性** | 予算外の別系統。ネタ合わせ/相方と過ごすで+1（上限20） | スコアに素点加算（+1相性=+1点） |
| **知名度** | 成長でなく経済。大会/フリーライブ/オファーで加算 | 決勝の人気補正（fameFinalBonus）で決勝ラインが下がる＝晩年の武器 |
| **ネタ鮮度**（将来） | neta_system_design（ネタ資産） | 別軸の伸び。上限到達後の主戦場になる想定 |
| **経済/体力** | バイト・休養 | 「伸ばす」から「整える（遠征費・体力・借金回避）」へ最適行動が移る |

設計意図: 上限を「手詰まり」でなく「**フェーズ移行の合図**」にする。演技4能力を伸ばしきったら、メンタル(安定)・相性(素点)・知名度(決勝補正)・ネタ鮮度へ戦略の軸が移り、谷口の台詞がその舵切りのトリガになる。「無意味な稽古」が「別の手を打つ局面」に変わる。

未決: 点灯閾値はsim後に確定／繰越年の文面分岐の要否（現案: 出す）。

## 5. GameCore接続アーキテクチャ

### 課題: 既存の `GameCareer.runYear` はポリシー注入型（同期ループ）で、UIの「週ごとに人の入力を待つ」と合わない

### 解法: WeekRunner（ステップ実行機）を追加し、runYear をその上に再実装する

【2026-07-04追記】WeekRunner本体は実装済み（`Sources/GameCore/WeekRunner.swift`・runYearには未接続の「追加のみ」）。実装で確定したAPI:

```swift
public struct WeekRunner<R: RandomSource> {
    public enum Phase {
        case tournamentDecision(TournamentSpec)      // 大会週: 出る?移動手段は? → resolveTournament(travel:)
        case freeAction(offer: OfferSpec?)           // 自由週: オファー抽選済み → resolveAction(_:)
        case gpRound(index: Int, name: String)       // GP回戦: 入力不要。演出後 resolveAuto()
        case gpRevival                               // 敗者復活: 入力不要
        case gpFinal                                 // 決勝: 入力不要（優勝なら resolveAuto() が yearDone を返す）
        case weekDone(WeekSummary)                   // 週の結果（results=大会結果一覧・state=週末処理後）
        case yearDone(YearOutcome)
    }
    public private(set) var state: GameState         // 年をまたぐ時は state/rng を次の WeekRunner に渡す
    public private(set) var rng: R
    public mutating func begin() -> Phase            // 次の週を開始（RNG消費は正典順）
    public mutating func resolveTournament(travel: Travel?) -> Phase   // nil=不出場
    public mutating func resolveAction(_ a: WeekAction) -> Phase
    public mutating func resolveAuto() -> Phase
}
```

- **正典性の担保**: `WeekRunnerGoldenTests` が CareerGoldenTests と同一の3年期待値（Python生成）でWeekRunnerを直接駆動する＝**runYearに触れずに等価証明できるテストを準備済み**。Macで両テストのgreenを確認したら、`GameCareer.runYear(policy:)` を「WeekRunnerを回してpolicyに聞くだけ」の委譲に書き換える（順序: ①swift test green確認〈CareerGolden＋WeekRunnerGolden〉 → ②runYear委譲 → ③再びgreen＝二重証明）
- ViewModel: `@Observable final class GameSession` が WeekRunner・GameState・演出キューを持つ。View層はPhaseのswitchで描画するだけ
- 保存: 週末ごとに GameState+進行状態をCodableでローカル保存（中断復帰）。GameState等へのCodable付与はこの段階で

## 6. コンポーネント規約

- 数値変化は必ず「差分ポップ」（+3 ↗ / -20 ↘）で見せる。絶対値の更新だけで済まさない
- フォント: 数字は等幅。ネタ帳S5のみ手書き風
- カラー: 体力は緑→黄(50)→赤(30)の3段（本番ペナルティ閾値と一致させ、色が警告の意味を持つ）

- SE: 行動決定1種・通過/敗退2種・週送り1種の計4種で開始

## 7-B. 検証①の合否基準【2026-07-04オーナー委任で確定・全て仮】

対象: S2〜S6が動く最初のビルドで、1年48週を**3周（144手）**セルフプレイする。

- **合格**: 3周を「やめ時を失って」完走できる。具体的には次の3つ全部:
  1. 2周目の途中で「義務でタップしている」感覚が出ない（出た週をメモ→3週連続で出たら不合格）
  2. 各周に1回以上「結果が気になって週送りが速くなる瞬間」がある（大会前後にあるはず。なければ緩急の設計ミス）
  3. 1周の所要が**8〜15分**に収まる（8分未満=見る物がない/15分超=テンポ悪い）
- **計測メモ**（プレイしながら3項目だけ記録）: ①義務感が出た週番号 ②行動選択で3秒以上迷った回数（迷い=考える楽しみ。1周で0回なら単調） ③スキップしたくなった演出名
- **不合格時の調整レバー（順番に試す）**: 会話・イベントの差し込み頻度UP（4〜6週に1本→3〜4週）→ 差分ポップの強調 → 大会演出の短縮 → 行動結果テキストのバリエーション追加。**ラインや数値バランスはこの段階では触らない**（体感問題を数値で直さない）

### S5 ネタ帳（ステータス）

役割: 能力5種・相性・記録の確認。**手書きノートの見た目**（§1・§6の「S5のみ手書き風フォント」）。§2-B の **成長ゲージ（伸びしろ）を能力欄の上に含める**——本画面の必須要素。

- **相性♥（/20 上限併記）**・**知名度★**: 能力とは別系統として区切る。相性は素点、知名度は決勝人気補正の武器（§2-B）。
- **記録**: 出場/通過/最高到達回戦・大会成績・獲得トロフィー（trophy_flavor 連携）。onboarding §4「記録は、ここから全部残る」の受け皿。
