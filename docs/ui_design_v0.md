# MVP画面設計 v0 — SwiftUI（1年48週版）

- 日付: 2026-07-04
- 位置づけ: MVP（1年版・検証①「週48手が飽きないか」）のUI仕様。GameCoreは実装済み（Career.swift）なので、本書は**画面・遷移・接続アーキテクチャ**を確定させる。数値・文言は全て【仮】。
- 上位規範: 週送り3秒ルール（1週=状況確認→行動選択→結果を無操作3タップ・3秒以内）／おすすめ行動表示の禁止／説明ダイアログ連発の禁止（improvement_proposals §4・onboarding §5）

## 2. S2 週メイン画面（核心）

```
┌─────────────────────────┐
│ 第12週 / 4月・春         [📅]  │  ← 週・季節。カレンダーへ
│ 💰 24.5万   ⚡ ▓▓▓░░ 55        │  ← 所持金・体力（数値+バー）
│ ★ 知名度 8   ♥ 相性 7          │
├─────────────────────────┤
│ （イベント/会話がある週はここに2〜4行）│  ← 谷口の一言・オファー提示
├─────────────────────────┤
│ [稽古 ▸]  [バイト ▸]  [休む ▸]     │  ← 3カテゴリ→展開で計11択
│ （オファー週: [仕事を受ける] が先頭に）│
└─────────────────────────┘
```

- **3秒動線**: カテゴリタップ→種目タップ→結果が同画面にトースト表示（画面遷移なし）→自動で次週へ。アニメは0.4秒以内・連打でスキップ可
- 行動ボタンには効果を**事前表示しない**（体力コストのみ小さく表示。効果量を隠すのが「考える楽しみ」——ネタ帳S5で育ちは確認できる）
- 有料稽古は所持金不足時グレーアウト＋「¥不足」
- 大会2週前からヘッダに「🏆 春新人賞Aまで あと2週」バッジ（エントリー確認モーダルは出さない＝仕様の2週前確認は表示のみ）

## 3. S3 大会画面（テンポの緩急はここで作る）

1. **入り**: 会場外観1カット＋大会名。大阪なら遠征選択（🚌夜行バス1万/体力-25 ⇄ 🚄新幹線3万）——**このゲーム唯一の「お金で安全を買う」瞬間**として選択肢を大きく
2. **本番**: 暗転→出囃子→**笑い波形メーター**（3〜5秒・ネタ本文は見せない）
3. **結果**: 溜め1秒→「通過」or「敗退」のスタンプ→**紙講評1行**（judge_design §8）→賞金・知名度の増分表示
4. GP回戦は1〜2を簡略化（毎年5回あるので15秒以内）。決勝週のみ finals_direction_v0.md のフル演出

## 4. S6 年次リザルト（MVP版）

1. **到達段階の縦の絵**: 回戦の階段（1回戦→…→決勝）に今年の到達点をマーク（improvement §1◎の簡易版）
2. 年間サマリ: 最終所持金・最低所持金・能力の伸び（開始→終了の矢印表示）・大会成績一覧
3. MVPは1年完結なので「この続き（10年のキャリア）は本編で」の予告1枚→タイトルへ

## 5. GameCore接続アーキテクチャ

### 課題: 既存の `GameCareer.runYear` はポリシー注入型（同期ループ）で、UIの「週ごとに人の入力を待つ」と合わない

### 解法: WeekRunner（ステップ実行機）を追加し、runYear をその上に再実装する

```swift
// 設計仕様（実装はMacで swift test が通ってから。runYearと同一セマンティクス必須）
public struct WeekRunner {
    public enum Phase {
        case tournamentDecision(TournamentSpec)      // 大会週: 出る?移動手段は? → resolveTournament(travel:)
        case freeAction(offer: OfferSpec?)           // 自由週: オファー抽選済み → resolveAction(_:)
        case gpAuto(round: Int)                      // GP回戦・決勝: 入力不要。演出後 resolveAuto()
        case weekDone(WeekSummary)                   // 結果（UIがトースト表示に使う差分）
        case yearDone(YearOutcome)
    }
    public mutating func begin() -> Phase            // 週の開始（RNG消費はここから正典順で）
    public mutating func resolveTournament(travel: Travel?) -> Phase
    public mutating func resolveAction(_ a: WeekAction) -> Phase
    public mutating func resolveAuto() -> Phase
}
```

- **正典性の担保**: `GameCareer.runYear(policy:)` を「WeekRunnerを回してpolicyに聞くだけ」に書き換える→ CareerGoldenTests（3年ビット一致）がそのままWeekRunnerの検証になる。**この書き換えはMacでgoldenが通る状態を確認してから行う**（順序: ①現状のswift test green確認 → ②WeekRunner追加とrunYear委譲 → ③再びgreen＝等価証明）
- ViewModel: `@Observable final class GameSession` が WeekRunner・GameState・演出キューを持つ。View層はPhaseのswitchで描画するだけ
- 保存: 週末ごとに GameState+進行状態をCodableでローカル保存（中断復帰）。GameState等へのCodable付与はこの段階で

## 6. コンポーネント規約

- 数値変化は必ず「差分ポップ」（+3 ↗ / -20 ↘）で見せる。絶対値の更新だけで済まさない
- フォント: 数字は等幅。ネタ帳S5のみ手書き風
- カラー: 体力は緑→黄(50)→赤(30)の3段（本番ペナルティ閾値と一致させ、色が警告の意味を持つ）
- ハプティクス: 大会結果と週送りのみ（連打操作に振動を付けない）
- SE: 行動決定1種・通過/敗退2種・週送り1種の計4種で開始
