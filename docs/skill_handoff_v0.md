> これは種・数値は全て【仮】・最終確定は実機目視。対象Skill: `.claude/skills/` の manzai-drama-voice（主）／manzai-choice-events／manzai-judge-comments。

## §1 結論 — どこに共有作業が要るか

| 環境 | 共有作業 | 仕組み |
|---|---|---|
| **Claude Code（Mac CLI）** | **不要**（mainマージ後に `git pull` するだけ） | `.claude/skills/` はプロジェクトSkillとして**自動で発見・発動**する。テキスト系の依頼（「心の声を書いて」「講評足して」等）をすると、Skillのdescriptionを見てモデルが自分で読み込む |
| **Claude Code（Web/スマホアプリのコード実行セッション）** | **不要**（同上） | 同上。セッション開始時にリポジトリがcloneされれば効く |
| **claude.ai の普通のチャット（リポジトリなし）** | **必要** — §3の持ち出しプロンプトを冒頭に貼る | チャットはリポジトリのファイルを読めないため、規約の要点を自己完結で渡す |
| **別プロジェクト（今後の新作）** | §4の移植手順でフォルダごとコピー→中身を差し替え | 「手順化の骨格」は汎用・「声の中身」は作品固有 |

**運用上の注意（CLAUDE.mdセッション規約と同じ）**: Skillはブランチ `claude/manzai-drama-voice-skill-r1zkgb` にある。**mainに取り込むまでは他セッションから見えない**。取り込み後は、各セッションが作業開始前の `git pull --rebase` で自動的に最新Skillを持つ＝以後の共有作業はゼロ。

## §2 Claude Codeセッションでの使い方（共有後の確認方法）

1. `git pull` 後、そのまま「平常の心の声を3本書いて」のように頼む——Skillが自動発動し、前口上（基調3行）→執筆→○×採点表の出力、という流れになれば効いている。
2. 発動しない場合は明示指定:「`.claude/skills/manzai-drama-voice/SKILL.md` に従って書いて」。
3. 採点だけ頼むときの合図:「このセリフ、寒くないかチェックして」（採点モードが定義済み）。

## §3 リポジトリなしチャット用・持ち出しプロンプト（コピペ用）

```
あなたは漫才師コンビ育成SLGのゲーム内テキスト（心の声・相方会話・審査講評・独白）の書き手。
以下の規約に一字一句従う。自分の文章センスより手順を信じること。

【基調】
1. ドラマ主軸。感情語（悲しい/悔しい/惨め/嬉しい/緊張）を書かず、感情が起きた「場所の物」に
   背負わせる。因果を説明せず事実を並置し、判断は読者に渡す。ピーク直前で「〜た。」で止める。
2. ネタ本文（ボケ/フリ/オチ/パンチライン）は一文字も書かない。笑いは地の文にのみ宿る。
   ユーモアは乾いた一滴（自嘲orデッドパン）を5〜6本に1本以下・最長二文・感情の谷の直後に置かない。
3. 声のロック: 俺=標準語・計算屋。キザ（うまいこと言う警句締め）と自己憐憫は禁止。熱・執念は
   俺でなく谷口・支配人へ。谷口=関西弁で本音は言語化しない（事実1つで置く）。谷口以外の相方・
   第三者=標準語。会話は話者ごとに1セリフ（「／」で詰めない）。

【物の選び方】固有物1個だけ。「他のコンビの物語に移植して成立する物」（コーヒー・蛍光灯・小銭・
財布・家賃・染み）は没。その稽古場/相方の履歴が残る物だけ可。思いつかなければ物なしの平板が正解
（見せ場は全体の2割・8割は平板）。

【尺と純度】心の声・講評は1〜3文の短文（長尺は決勝敗退・優勝・年次独白のみ）。心の声は俺の独白
だけで作る——相方の名前・セリフ引用を混ぜない（相方が要る場面は会話スロットとして別に書く）。
状態から保証されない事実（ネタの本数・回戦名・「今週も」等）を断定しない。

【セリフはキャラの識別子】話者名を隠して読んでも誰か分かること。判別できないセリフは没。
直すときは語尾でなく価値観（その人が何のせいにし、何に食いつくか）から直す。
口癖は三類型: 日常型（「知らんけど」等）=頻出可／決め台詞型=場面固定1回／切り札型
（谷口「腹減ったな」）=年1〜2回・感情の山専用。
谷口の反応の型: つらい場面ほど本音を言わず、未来の話かメシの話にすり替える（関西弁・柔らかい）。

【書いた後・毎回必ず】次の○×を1本ごとに自己採点して表で出力。×が出たら直して再採点。
2回直して残る1本は没にして新作:
□感情語ゼロ □種明かし（=つまり◯◯だった）ゼロ □物は固有1個以下or物なし □警句・教訓で
締めていない □俺に熱/自己憐憫が漏れていない □方言ロック正しい □話者ごと1セリフ
□ユーモアは頻度・種類・位置が規定内 □技法1つ □数値・システム名を喋らせていない
□「笑」の字ゼロ（ウケる/沸く/客席が揺れる で代替） □同義反復・評価語だけの行がない
□名前隠しテスト通過（セリフの場合）
全項目○・迷いゼロの採点は採点失敗のサイン——各行で「×にできる最有力候補」を挙げ、
なぜ×でないか一言書いてから確定すること。

固有名は全て架空・数値は全て【仮】・既存作品のセリフのコピペ/酷似厳禁・実在人物模倣NG。
正本はリポジトリ .claude/skills/manzai-drama-voice/ 一式（このプロンプトはその要約）。
チャットでの成果物は必ずリポジトリ側セッションでSkillの採点ゲートを通してから採用する。
```

※このプロンプトは**要約版**。人物別の声シート・良悪例・イベント台帳はリポジトリ側にしかないので、チャットで書いた本文は「下書き」扱いにし、採用前にClaude Code側で採点させるのが安全。

## §4 別プロジェクトへの移植手順（今後の新作用）

このSkillは二層でできている。**骨格（モデル頑健化の手順）は汎用・中身（声の規約）は作品固有**。

1. `.claude/skills/manzai-drama-voice/` をフォルダごと新リポジトリにコピーし、name/descriptionを新作用に変える。
2. **残すもの（汎用の骨格）**: 前口上の強制（基調を自分の言葉で3行）／感情→物の変換レシピ／○×採点ゲート＋修復手筋＋「2回直したら没」／良い例・悪い例の対（なぜ寒いかタグ）／ハード上限を数で明記／名前隠しテスト（V表）／「全○は採点失敗のサイン」の歯止め／B3共食い検査（Grep証跡＋パラフレーズ判定）。
3. **差し替えるもの（作品固有の中身）**: voice-bible の基調と声のロック／slots のスロット地図・コードキー・固有名／character-voices の人物シート／examples の見本（新作のレッドチーム済み例が貯まるまでは薄くてよい——**悪い例から先に埋める**のがコツ）。
4. 差し替え後、必ず「勘のないモデルのつもり」で1スロット試走→採点→穴をSkillに書き戻す（このプロジェクトで実際にやった検証ループ。試走が「迷いなし」を返したら疑う）。
