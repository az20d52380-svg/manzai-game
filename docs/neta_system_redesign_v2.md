## 0. 結論（3行）

1. **ネタは「年始に1つ選ぶ型」ではなく、`ネタ作り` で1本ずつ生まれる個体**。`ネタ見せ会`/`フリーライブ` で客前にかけると完成度が上がり、**持ちネタ帳（レパートリー）に貯まって年を跨いで持続**する。何年も前のネタを今年の大会で選んでもよい。型は年の選択でなく個体の属性（`Neta.kata`）。**却下された「年始に型を選ぶ」の再混入は無い**（批評1チェック5合格）。
2. これは**新機構ではなく、既存の稽古アクション（`GameConfig.swift:151-155`）・大会/GP（`Calendar.swift`）・採点式（`GameEngine.swift:145-158`）・見せ札演出（`FinalsPresentationView.swift`）・ネタ帳UI（`NotebookView.swift`）を"ネタ個体"で束ね直すだけ**で成立する。GameState へのフィールド追加は `exp*` 追加（`GameState.swift:30-38`）と同型＝**Codable自動・golden不変**（`CodableTests.swift:11`／全引用は本改訂で実ファイル照合済み）。
3. **Phase 0＝ネタを作り・貯め・"どれで出るか選ぶUIの土台"までを、合否スコアに一切効かせず golden 完全不変で立てる**。ただし**モデルの核である「客の反応で磨く」「反応の良いネタを選ぶ意思決定」は、当たり外れ＝新しい乱数か、スコア寄与を要するため Phase 0 では原理的に立ち上がらない＝Phase 1（規律A・4.8相談・sim必須）が必須**（v1の「Phase 0 で核が立ち上がる」は過大主張だったので本版で訂正・§0-補／§6/§7）。7審査員×型の相性は Phase 0 では**講評/見せ札tiltの演出材料（非スコア）**に留める。

> **§0-補【v1からの最重要訂正・3批評の合流点】**: v1 は「作る・磨く・貯める・選ぶ」が Phase 0 で完全に立ち上がると繰り返したが、**実際に Phase 0 で立つのは「作る・貯める・選ぶ器（UI）」まで**。「反応で磨く」は `buzz` が当たり外れの情報を持って初めて成立し（批評1 ズレ1・批評3 欠落3/違和感2）、「選ぶ意思決定」は選択が結果に効いて初めて成立する（批評1 ズレ2・v0 §6-B 実測で "賢い選択 vs ランダム選択" は±0.5pt＝ノイズ帯と実証済み）。**この2つはどちらも Phase 1 が前提**。よって決点（§9）で「Phase 1 に進むか」を"任意"から**"モデル核の実装に必須"**へ格上げし、`buzz` に客層依存（当たり外れ）を入れるかを**新決点6**として顕在化させる。

## 1. 実在漫才の要点（調査1〜5の要約・設計が背負う事実）

本設計が忠実再現すべき、実在漫才の**構造的事実**だけを抜く（出典は §9）。★＝v1で薄かった／欠落し、本v2で回収する事実（批評3）。

| # | 事実 | 設計への要請 |
|---|---|---|
| A | ネタは**1本ずつ作る**。「テーマ→ボケ大量出し→(他人に見せて)選別→構成→台本化」で1〜2週間（調査1 §2） | `ネタ作り` は**1本の個体を生む/伸ばす**行為。抽象値+3では欠落 |
| B | **新ネタは"おろす"（初披露）→劇場・営業で場数→客反応で削り直す**。「漫才はドラクエ、やればやるほど経験値」（調査2 §2/§6） | ライブでかけるほど完成度が上がる。★**"おろす"（初披露）は一回性の節目**＝ `stageCount:0→1` を無味な+1にしない（欠落1） |
| C | **客の反応でウケ/スベりが可視化**され、それが改稿の羅針盤（調査2 §3）。会場・客層で反応が変わる（場数=実験）。**スベりは磨きの一次情報** | ライブの手応え＝反応を個体に記録。★**"反応の良し悪し"が磨きの方向を決める＝反応→磨きの因果**が要る（欠落3・違和感2）。これは乱数/スコアを要する＝Phase 1 |
| D | **持ちネタは芸歴で貯まる資産**。**古いネタの再演が主流**（アンタッチャブル/タカトシ=20年超前を更新して現役）。良ネタの寿命=十数年〜20年超（調査3 §3/§4） | ネタは年を跨いで持続。作成年を記録。★**再演は肯定的な行為**（"寝かせた勝負ネタが今の客に刺さる"）＝連投ペナルティ一辺倒にしない（欠落6） |
| E | **鉄板3本あれば一流**（上岡龍太郎）＝数より質。だが**レパートリー総数は芸歴で数十本に積む**（中川家=30年で多数蓄積）。この2つは別軸（調査3 §2/§6） | ★**「同時に磨く鉄板＝少数」と「保管庫の総数＝多数・年跨ぎ資産」を分離**（批評1 ズレ3）。4本キャップで倉庫を絞ると「20年前のネタを今やる」＝Dと衝突 |
| F | **大会は場でネタを使い分ける**。予選=手堅い勝負ネタ/決勝=2本制（1本目=空気掌握/2本目=温存の最強 or 意表）。準決勝マニアック→決勝万人ウケで**載せ替える**。**尺（持ち時間）が第一の制約**（1回戦2分/準々〜決勝4分・「4分と2分は中距離走と100m走ほど違う」）（調査4 §0/§2/§3/§4） | 大会でどのネタを出すか選ぶ。★**尺＝ネタ選びの第一関門**（欠落2）。★**予選と決勝で客層が違うから載せ替える動機**（欠落4）。同系統で同じネタ連投は目減り |
| G | **型は2軸が混在**。演じ方軸（しゃべくり/コント＝1本1型が自然）＋技法軸（伏線回収・天丼・つかみ＝本来"部品"で1本に併存しうる）。**磨く過程で勝負どころが移ることもある**（システム漫才は演じるたび変わる）（調査5 §3/§4） | 型はネタが持つ1属性（年単位の選択ではない）。「最も勝負している要素」1つ。★ただし**作成時に凍結せず**、磨く中で1度組み替え可（批評1 ズレ4・批評3 違和感4） |

> **核**: 「作る → ライブで磨く（反応が羅針盤・完成度↑）→ 持ちネタに貯まる（年跨ぎ持続・古いネタも再演）→ 大会で反応の良いネタを選んで出す（尺と客層で使い分け）」の**持続ループ**。型は年の選択でなく個体の可塑的属性。**このうち"反応で磨く""選択が効く"は Phase 1 が担う**（§0-補）。

## 2. ネタのデータ構造【仮】

### 2-1. `Neta` 個体（GameCore・新規 `Neta.swift`）

`exp*`（`GameState.swift:30-38`）や `ChoiceEventEffect.swift`（実在確認済み）と同じ**純Swift・RandomSource非依存・Codable**。数値は全て【仮】。

```swift
// Neta.swift（新規・GameCore）— RandomSource を一切呼ばない純データ＋純関数。
// golden非干渉: このファイルのどの関数も乱数を引かない・GameEngine.perform の式に触れない（Phase 0）。
public enum NetaKata: String, Codable, CaseIterable {   // 型＝ネタが持つ1属性（年の選択ではない・§1-G）
    case 王道しゃべくり   // 会話の純度・間
    case 関係性         // 二人の歴史・掛け合い
    case 伏線回収        // 多層構成
    case リターン        // 天丼・帰る場所（反復）
    case 非定型         // 新しい型の発明（実験）
    case 瞬発          // 畳みかけ・勢い押し
    case 華先行         // つかみ・存在感で持つ
    // 注: judge_preference_research_v0.md §5／multiyear §2-3 の7型と同一。旧v0の5型を7型に更新（§8）。
}

public enum NetaLength: String, Codable, CaseIterable {  // ★尺＝実在の第一制約（§1-F・批評3 欠落2）
    case 短尺   // 詰めて2分で成立する型
    case 中尺   // 3分で映える
    case 長尺   // 4分の展開・伏線が生きる型
}

public struct Neta: Codable, Identifiable {
    public let id: Int                 // 生成連番（セーブ安定・表示ソート用）
    public var name: String            // 題材名（自動生成 or プレイヤー命名・一般名詞のみ／§9-決点3）
    public var kata: NetaKata          // 型＝1属性。★let→var（磨く中で1度だけ組み替え可・§3-1補／批評1 ズレ4・批評3 違和感4）
    public var lengthFit: Set<NetaLength>  // ★このネタが映える尺（純データ・Phase 0は表示のみ／§4-1補）
    public var polish: Double          // 完成度 0..100（稽古・ライブで上がる。下がらない資産）
    public var buzz: Double            // 手応え 0..100（ライブでの反応の移動平均・§3-2）。★Phase 0は決定論=polish従属（限界は§3-2補で明示）
    public let bornYear: Int           // 作成年（芸歴年・§1-D「古いネタも再演」の証拠）
    public var stageCount: Int         // 通算ライブ数（"漫才はドラクエ"の経験値・§1-B）
    public var isDown: Bool            // ★おろし済（初披露を経たか・§1-B/批評3 欠落1）。false=未おろし
    public var exposure: Double        // ★露出度（大会で使うほど+。"広く知られた度"の表示・§3-2補/批評3 方針D）。Phase 0は非スコア表示のみ
    // 戦績（§5-3 ネタ帳の"育った実感"用・非スコア）
    public var record: [NetaStamp]     // 「3回戦・通過」等のスタンプ列
    public var lastUsedTaikaiYear: Int?  // 同系統での連投目減り判定に使う年（§4-2）

    public init(id: Int, name: String, kata: NetaKata, lengthFit: Set<NetaLength>, bornYear: Int) {
        self.id = id; self.name = name; self.kata = kata; self.lengthFit = lengthFit
        self.polish = 30; self.buzz = 0; self.bornYear = bornYear
        self.stageCount = 0; self.isDown = false; self.exposure = 0
        self.record = []; self.lastUsedTaikaiYear = nil
    }

    // 派生状態（純関数・表示専用・非スコア）——§5-3で「鉄板」「再演」バッジに使う。
    public func isTeppan(config: GameConfig) -> Bool {   // ★鉄板＝高完成度×場数×安定（§1-E/批評3 欠落5）
        polish >= config.netaTeppanPolish && stageCount >= config.netaTeppanStage && buzz >= config.netaTeppanBuzz
    }
    public func isRevival(currentYear: Int, config: GameConfig) -> Bool {  // ★数年寝かせたネタの再演（§1-D/欠落6）
        currentYear - bornYear >= config.netaRevivalYears
    }
}

public struct NetaStamp: Codable {     // 戦績スタンプ（表示専用）
    public let year: Int
    public let stage: String           // "GP3回戦" 等（StageResult.name をそのまま）
    public let passed: Bool
}
```

- **完成度 `polish` と 手応え `buzz` を分ける理由**（調査2 §3）: `polish`=磨き込みの累積（下がらない資産）。`buzz`=直近のライブでの反応。**ただし v1 は「別軸に立てた」と謳いながら `buzz` を `polish`/`power` の単調増加関数にしてしまい、"低反応という情報"が生まれず、別軸が算出式で死んでいた**（批評1 ズレ1・批評3 欠落3/違和感2）。**Phase 0 の `buzz` は決定論の"手応え表示"に留まり、真に別軸（客層で当たり外れが出る＝磨きの羅針盤）になるのは客ムラ乱数を入れる Phase 1＝新決点6**（§3-2補・§9）。この限界を隠さず明示する。
- **`isDown`（おろし済）**: 旧v0 §1.2 にあった「下ろし済フラグ」を v1 が落としていたので復活（批評3 欠落1）。`stageCount:0→1` の初ライブ＝「ネタおろし」を一回性の演出にする（§3-2補）。純データ・Codable自動・非スコア。
- **`lengthFit`（尺）**: 実在で最も具体的な戦略要素「2分用に詰めたネタ／4分用に展開のあるネタ」（§1-F）。Phase 0 は**選択時に"この回戦の尺に合う/合わない"を表示するだけ**（合否には効かせない）。スコアに"尺ミスマッチ減点"を乗せるのは Phase 1（§6-2・要sim）。
- **`kata` を `var` に**（`let`から緩和）: 個体属性化は正しい（再混入ではない）が、実在は「磨くうちに勝負どころが移る」（システム漫才）。作成時凍結は"作り込む中で性格が変わる"を塞ぐので、**大改稿で1度だけ型を組み替え可**にする（§3-1補・稀な操作・純データ・非スコア）。

### 2-2. 持ちネタ帳（GameState 拡張・golden安全）— アクティブ枠と保管庫の二層

`GameState`（`GameState.swift:4-52`）に**フィールドを足すだけ**。`exp*` 追加（同 30-38）と完全に同型＝Codable自動・`CodableTests.swift:11` の往復に1行足すだけで担保。**perform（`GameEngine.swift:145-158`）はこの配列を読まない＝golden不変**。

```swift
// GameState.swift への追加（Phase 0）— exp* と同じ「表示・資産の器」。perform は参照しない。
public var netas: [Neta] = []          // ★アクティブな持ちネタ（少数・磨き対象＝鉄板枠。§1-E前段）
public var archivedNetas: [Neta] = []  // ★保管庫（多数・年跨ぎ資産＝倉庫。いつでも大会に呼び戻せる。§1-E後段/批評1 ズレ3）
public var nextNetaID: Int = 0         // 生成連番の採番（決定論・乱数非依存）
public var selectedNetaID: Int? = nil  // 大会に「今かける」ネタ（自由週にデフォルト=前回踏襲・§4）
public var selectedNetaID2: Int? = nil // 決勝の2本目（§4-2）
```

- ★**二層に分離**（v1の「4本枠・5本目で1本封印」を改める・批評1 ズレ3/批評3 違和感5）: v1 は「4本キャップ・5本目作成時に1本を封印（`archivedNetas` へ玉突き）」だったが、これは実在D「20年前のネタを今やる」＝古いネタが**貯まって残る**要件と直接衝突する（新作のたびに古い鉄板が押し出される）。**"同時に磨く鉄板は少数"（アクティブ枠）と"レパートリー総数は多数"（保管庫）を混同していた**。
  - **アクティブ枠 `netas`**＝少数（**枠数は決点2で確定**・目安3〜5本）。ここが磨き（`ネタ見せ会`/`フリーライブ` の対象）と週の視野。枠が埋まったら1本を**保管庫へ退避**（削除でも封印でもない）。
  - **保管庫 `archivedNetas`**＝**総数に上限を置かない**（芸歴で貯まる）。**いつでもアクティブ枠に呼び戻して大会に出せる一級市民**（§5-3で古いネタ再演をUI上で主役に）。玉突きで"供養"されない。
- **なぜ配列で golden 安全か**: `netas`/`archivedNetas` は `perform` の入力（`jitsuryoku + s.compat + roll + penalty`・`GameEngine.swift:158`）に一切入らない。追加は「セーブに乗る器」を増やすだけ。乱数消費（perform の2draw＝`GameEngine.swift:149-150`／rollOffer＝`139-140`／injury＝`WeekRunner.swift:228`）の**順序も回数も変わらない**。
- ★**init は新フィールドを代入しない**（批評2 C-2）: `GameState.init(config:)`（`GameState.swift:40-52`）は `exp*` 同様に新フィールドへ代入しない＝**プロパティ既定値（`= []`／`= 0`／`= nil`）に委ねる**（Swiftが保証）。実装者が `init` に足して差分を生まないよう、**init には書かない**方針を明記。

## 3. アクション接続 —「作る」「磨く」を既存5稽古に接地

既存の稽古5種（`GameConfig.swift:151-155`／UIカタログ `CommandData.swift`）の**能力効果は据え置き**、そこに**ネタ個体への作用を"重ねる"**。v1 §3 の方針を、二層・尺・おろし対応に更新。

### 3-1. 「作る」＝`ネタ作り`（`GameConfig.swift:151`・現 発想+3/センス+1）

| 現行（据え置き） | ネタ個体への作用（新・Phase 0） |
|---|---|
| 発想+3・センス+1（会計移設後は粒） | **アクティブ枠に空きがあり選択中ネタが無ければ新ネタ生成**（`polish=30`・型と`lengthFit`を選択・`nextNetaID` を採番・`isDown=false`）。**あれば改稿＝`polish += 8`**【仮】 |

- **型と尺の選択タイミング**（§1-G「1属性」の担保）: **新ネタ生成時のみ**、型を1つ＋映える尺を選ぶ（「このネタで最も勝負する要素は?」「何分用か?」）。**年の選択ではない・ネタごと**。旧Fable案（年始に1つ）との決定的な違いはここ。
- **UI**: `ネタ作り` カードをタップ→（アクティブ枠に空きがあり選択中ネタが無い/新規を選んだ場合）**型＋尺の小シート**を1枚挟む→生成。改稿の場合は選択中ネタに +polish。**3秒動線を壊さないため、デフォルトは「選択中ネタを改稿」**（型/尺シートは新規時のみ）。

- ★**§3-1補【型の組み替え】**（批評1 ズレ4・批評3 方針G）: 既存ネタに `ネタ作り`（改稿）を**規定回数以上**重ねた大改稿で、**「このネタの勝負どころを変える」**を稀な操作として提示（`kata` を1度だけ変更可）。純データ・非スコア・RNG非消費。実在の「磨くうちに伏線回収型に化ける」を最小限で回収。デフォルトは型不変（誤操作防止）。

### 3-2. 「磨く」＝`ネタ見せ会`／`フリーライブ`（客前でかける）

実在の核（調査2 §2/§6「劇場でかけるほど完成度が上がる／客反応が羅針盤」）を、**この2アクションに背負わせる**。**ただし v1 の "ライブに出れば polish は自動加算・buzz は横で表示"は因果が逆**（反応→磨きになっていない）＝この構造的限界を §3-2補で明示し、真の"反応で磨く"は Phase 1 に置く。

| アクション | 現行（据え置き） | ネタ個体への作用（新・Phase 0） |
|---|---|---|
| `ネタ見せ会`（`GameConfig.swift:152`・表現+6/メンタル+3・有料80,000） | 舞台稽古＝表現・メンタル | 選択中ネタに **`polish += 12`・`stageCount += 1`**。初回なら **`isDown=true`（おろし）**。この回の手応えを算出（下記）→ `buzz` を移動平均で更新【仮】 |
| `フリーライブ`（`GameConfig.swift:155`・華+3/表現+1・無料・知名度+1） | 客前で場数 | 選択中ネタに **`polish += 6`・`stageCount += 1`**・初回なら **`isDown=true`**・手応え算出→`buzz` 更新。営業=数を踏む場（安いぶん磨きは緩め） |

- **手応え（`buzz`）の算出は決定論・純関数＝golden非干渉**（Phase 0の生命線）:

```swift
// Neta.swift — ライブ1回の手応え。RandomSource を呼ばない純関数（golden非干渉）。
// 実在: 出来（今の実力）× ネタの完成度で沸きが決まる（調査2 §3）。★客層差＝当たり外れは Phase 0 では入れない（下記補）。
public static func liveBuzz(state s: GameState, neta: Neta, config: GameConfig) -> Double {
    let power = GameEngine.jitsuryoku(s, config: config)   // 参照のみ・perform の乱数には触れない（GameEngine.swift:51）
    let raw = power * config.netaBuzzPowerW + neta.polish * config.netaBuzzPolishW
    return GameEngine.clamp(raw, 0, 100)
}
// buzz の更新は移動平均（直近のライブほど効く）:
//   neta.buzz = neta.buzz * (1 - α) + liveBuzz(...) * α   （α=0.4【仮】）
```

  - **なぜ乱数を引かないか**: ここで `rng.nextUniform()` を引くと**消費順が動き 3年 golden が壊れる**（`WeekRunner.swift:228` の injury 抽選と同じ位置＝RNGは消費順が正典・規律A-2）。Phase 0 は実力×完成度の決定論値にする。
  - **実装位置**: `applyTraining`（`GameEngine`）は選択中ネタIDを知らないため、**ネタ作用は WeekRunner 側の "行動確定の直後・RNG非消費の区画" に置く**——`applyAllocation`（`WeekRunner.swift:338`）/`applyEventEffects`（同 `330`）と**同じ"権威stateへの純適用"経路**を新設（`applyNetaWork(training:)`）。この区画は**乱数を一切呼ばないことがコメントで保証**されている領域（`WeekRunner.swift:328`「RandomSource を一切呼ばない＝乱数消費順は1ビットも動かない＝3年 golden 不変」）＝そこに相乗りする（批評2 で "設計が主張する以上に強く裏取りされている"と確認済み）。

- ★**§3-2補【本版最重要・"反応で磨く"の構造的限界を明示】**（批評1 ズレ1・批評3 欠落3/違和感1/違和感2/違和感3・方針D）:
  - **因果の問題**: Phase 0 の設計は「ライブに出れば `polish` は固定加算・`buzz` は算出して横に表示」。実在の因果は**反応→磨き**（スベった箇所を見て削る）だが、Phase 0 では `buzz` の高低が `polish` の上がり方を変えない＝**"反応を見て磨く"ループが機構化されていない**。`buzz` が `polish`/`power` の単調増加関数である以上、**"低反応＝スベり"という情報が生まれない**（`liveBuzz` は右肩上がりのみ）。したがって Phase 0 で立つのは**"客前に出た回数で `polish` が貯まる"**であって、**"反応で磨く"ではない**。この誇張を隠さない。
  - **`polish` 単調増加＝劣化なし**（批評3 違和感1）: 実在は「広く知られたネタは決勝で出しづらい＝新鮮さが消耗する／勝負ネタの在庫管理」（調査4 §4）。`polish` が一方的に上がるだけだと"劣化しない最強ネタ"が生まれ、温存戦略が成立しない。**折衷**: `polish` は資産のまま据え置き、**`exposure`（大会で使うほど+）で"擦り切れ度"を表示のみ**（非スコア・§5-3）。スコアで温存を効かせるのは Phase 1。
  - **改稿の粒度**（批評3 違和感3）: `polish += 8/12` の一律加算は"経験値バーを溜める"作業で、実在の"削って直す"手仕事ではない。**Phase 0 の抽象化としては許容**（局所改稿のモデル化は範囲外）だが、演出（`manzai-drama-voice`）で「今日はここが転んだ」の質感を補う。
  - **結論**: **真に"客の反応で磨く"（当たり外れ＝スベりが出て、それが選別と改稿の羅針盤になる）を実装するには、`buzz` に客層依存＝新しい乱数が要る＝Phase 1／新決点6**（§9）。Phase 0 は「作って場数で `polish` を貯める土台」までと正直に位置づける。

- ★**§3-2補2【ネタおろし＝一回性イベント】**（批評3 欠落1・方針A）: `isDown:false → true`（初ライブ）を**「ネタおろし」として特別演出**（`manzai-drama-voice`）。ネタ帳で「未おろし／おろし済」を区別表示。初披露の手応えを**一拍大きく見せる（表示のみ・非スコア）**。実在で最もドラマがある「新ネタを初めて客前にかける緊張」を体験に立てる。RNG非消費・golden不変。

### 3-3. `ネタ合わせ`／`ランニング・サウナ` は据え置き（ネタ非依存）

- `ネタ合わせ`（`GameConfig.swift:153`・センス+3/相性+1）＝「ネタより二人の呼吸」の受け皿。ネタ個体には触らない。
- `ランニング・サウナ`（同 `154`・メンタル+6）＝体調・メンタル。ネタ非依存。

> **接続の要旨**: `ネタ作り`＝生む/改稿（polish・型組み替え）。`ネタ見せ会`/`フリーライブ`＝**客前でかける＝手応え算出＋polish＋おろし**（実在の"場数"）。`ネタ合わせ`/`ランニング`＝ネタ非依存の土台。**全て既存アクションに乗り、能力効果は据え置き。ネタ作用は RNG非消費の純適用区画に置く＝golden不変**。**ただし"反応で磨く"の核は §3-2補のとおり Phase 1**。

## 9. 要オーナー決点

> **決点1（型の数・7 vs 5／型の可塑性）**: 本書は**7型**（王道しゃべくり/関係性/伏線回収/リターン/非定型/瞬発/華先行）＝審査員相性表（§4-4）と1対1。**7でよいか／5に減らすか**。推奨: 7。加えて★**型を大改稿で1度組み替え可**（`var kata`・§3-1補）とするか。推奨: 可（実在の可塑性・稀な操作で誤操作は防ぐ）。

> **決点2（持ちネタの二層・アクティブ枠数 vs 総保管数）**: v1 の「4本キャップ」は実在D（古いネタ再演）と衝突するため、★**アクティブ枠（少数・磨き対象）と保管庫（多数・年跨ぎ・呼び戻せる）の二層**に改めた（§2-2）。**アクティブ枠を何本にするか（3/4/5）／保管庫は無上限でよいか**。推奨: アクティブ3〜4本＋保管庫無上限（鉄板は少数精鋭・レパートリー総数は芸歴で貯まる、を両立）。

> **決点3（ネタ命名・自動 vs プレイヤー）**: 題材名を**自動生成**（一般名詞リスト・実在ネタと偶然一致しても問題ない語に限定）にするか、**プレイヤー命名**を許すか。推奨: 自動をデフォルト＋任意で改名可（`Neta.name` は var）。命名リストは `name_generator` と別管理。

> **決点4【格上げ・最重要】（Phase 1 に進むか＝ネタを勝敗に効かせるか）**: ★v1 は"任意"扱いだったが、**§0-補のとおり "反応で磨く""選択が効く" というモデルの核は Phase 1 でしか立たない＝Phase 1 はモデル実装に必須**（批評1 最優先）。§6-2(a)(b)＝完成度・手応えのスコア補正＋2本制ペナ＋尺ミスマッチ＋再演/連投補正を**入れるか**。入れると「反応の良いネタを選ぶ」が本当に勝率に効く（実在Fの戦略が生きる）。代償は**規律A（golden全再生成）＋sim較正**（`exp_neta.py` 実測＝係数0.04で中央値±0年・v0 §6-B `docs/neta_system_design_v0.md:88-95`）。推奨: **Phase 0 を先に出し体感を見てから即 Phase 1（P1-a）へ**——Phase 0 単独では"器"に留まる点を承知の上で。

> **決点5（型×審査員相性をスコアに乗せるか＝§6-2(c)/P1-b）**: `judge_design §0`「審査は総合点・違いは表示のみ」の**格上げ**（multiyear §2-6 と同一の未決）。推奨: **当面乗せない**（表示・講評で"読み"の8割は成立し、golden全再測＋endgame/dynasty 再アンカーの代償が重い）。**オーナー判断待ち**。

> ★**決点6【新設】（`buzz` に客層依存＝当たり外れを入れるか＝P1-c）**: v1 は「buzz を別軸に立てた」と謳いつつ算出式が決定論で退化していた（§3-2補）。**実在の"ライブ磨き"の核（同じネタが会場で沸いたりスベったりする＝それが磨きの羅針盤・載せ替えの動機）は、`buzz` に客層依存＝新しい乱数を入れて初めて立つ**（批評3 方針F）。だが**新規乱数＝消費順が動く＝golden全再生成＋規律A・4.8相談が必須**（§6-2(d)・🚫止めて相談の正しい対象）。推奨: **まず入れない**（golden の代償が最も重く、Phase 0/P1-a で"読み"の体験は成立する）。ただし**「決定論の buzz では"反応で磨く"は原理的に立たない」ことを承知した上で判断**。入れると決めたら P1-c として規律A第1段フルセット。
